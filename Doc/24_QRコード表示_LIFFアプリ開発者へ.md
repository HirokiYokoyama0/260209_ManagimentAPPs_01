## 0. LIFFエンジニア向け よくある質問

**Q. ダッシュボード側のQRコード生成コードはどこにありますか？**  
**A.** 管理ダッシュボード（本リポジトリ）の **`app/admin/qr/page.tsx`** です。`QRCodeSVG` で2種類のQRを表示しています。

**Q. QRコードのペイロード（中身のJSON）は何ですか？**  
**A.** 次の2種類です（**スタンプ個数に統一。ポイントは廃止**）。

| 種類 | ペイロード文字列 |
|------|------------------|
| 優良患者様用 | `{"type":"premium","stamps":10}` |
| 通常患者様用 | `{"type":"regular","stamps":5}` |

**Q. 画面上でQRの中身を直接確認できますか？**  
**A.** はい。管理画面の「テストQR」ページでは、各QRの**直下にペイロード文字列**を表示しています。スキャン前に目視で確認できます。

**Q. `POST /api/stamps` が呼ばれているが、通常患者用QRでも1スタンプしか付かない**  
**A.** ダッシュボードのQRは上記のとおり **stamps:10（優良）/ stamps:5（通常）** を出しています。問題は **LIFF側でQRをデコードしたあと、APIに渡す値を「読み取った `stamps` の値」にしているか** です。APIが「常に1スタンプだけ加算」する実装になっていると、通常患者用（5スタンプ）を読んでも 1 しか付きません。**APIのリクエストBodyに、QRの `stamps` をそのまま渡すよう実装を確認してください。**

---

## 1. 概要

- 患者が**来院時**に、院内に設置されたQRコード（またはテスト用として管理画面に表示されたQR）を、**LIFFアプリ内のカメラで読み取る**。
- 読み取った内容に応じて、**スタンプ付与API** を呼び出し、`stamp_history` に1件登録する。これにより `profiles.stamp_count` 等が既存トリガーで自動更新される。
- 本ドキュメントでは、**QRコードのデータ形式**と、**読み取り後にLIFF側で呼ぶAPIの仕様**を定義する。

---

## 2. QRコードのデータ形式（ペイロード）

**※ ポイント概念は廃止済み。スタンプは「個数」に統一。** ペイロードは **stamps**（付与スタンプ数）を使用する。

管理ダッシュボードの「テスト用QRコード」タブで表示する2種類のQRは、次の **JSON文字列** をそのままエンコードしたものとする（改行なし・UTF-8）。

### 2.1 優良患者様用（10スタンプ/回）

```json
{"type":"premium","stamps":10}
```

- **type:** `premium` … 優良患者様用
- **stamps:** `10` … 1回のスキャンで付与する**スタンプ個数**（10個）

### 2.2 通常患者様用（5スタンプ/回）

```json
{"type":"regular","stamps":5}
```

- **type:** `regular` … 通常患者様用
- **stamps:** `5` … 1回のスキャンで付与する**スタンプ個数**

### 2.3 共通

- 将来的に本番用QRで `qr_code_id`（重複防止用）や `expires_at` などを含める場合は、LIFF側でそれらをパースし、APIに渡す想定。**テスト用では上記2種類の固定ペイロードのみでよい。**
- **旧仕様の `points` は使用しないこと。** LIFF側は必ず **stamps** を参照し、APIには付与する**スタンプ個数**を渡すこと。

---

## 3. LIFF側の処理フロー

1. **カメラ起動**  
   LIFFアプリ内でカメラを起動し、QRコードを読み取る（LIFF SDK または HTML5 の `getUserMedia` + 任意のQRデコードライブラリ）。

2. **デコード**  
   読み取った文字列が上記JSON形式であればパースする。**`type` と `stamps`（スタンプ個数）を取得。** 不正な形式の場合はエラー表示し、再スキャンさせる。**`points` は廃止のため参照しないこと。**

3. **スタンプ付与APIの呼び出し**  
   - **エンドポイント:** LIFF側で利用しているスタンプ登録API（例: `POST /api/stamps`）に、**QRの `stamps` の値を渡す**こと。1回のスキャンで「何スタンプ付与するか」をAPIが受け取り、その数だけスタンプを加算する実装とする。  
   - **リクエスト例（JSON）:**
     ```json
     {
       "type": "premium",
       "stamps": 10,
       "qr_code_id": "optional-unique-id-for-dedup"
     }
     ```
     優良患者様用の場合は `"type":"premium","stamps":10`、通常患者様用の場合は `"type":"regular","stamps":5` をそのまま渡す。
   - **認証:** 呼び出し元が「どの患者（LINEユーザー）か」を識別する必要がある。LIFFの `getProfile()` で取得した `userId`（LINEのユーザーID）を、ヘッダーまたはBodyに含める。API側で `profiles` の `id` または `line_user_id` と突き合わせ、該当患者の `stamp_history` にINSERTする。
   - **重複防止:** 同一QRを短時間に連打されないよう、`qr_code_id` をリクエストに含め、サーバー側で「同一 `qr_code_id` + 同一ユーザー」の連続登録を制限する設計が望ましい。テスト用では `qr_code_id` を省略または固定でも可。

4. **レスポンス**  
   - 成功: 200 + 付与後のスタンプ数など（任意）。LIFF側で「スタンプが付与されました」と表示。
   - 失敗: 4xx/5xx + エラー内容。LIFF側でメッセージ表示し、必要なら再スキャンさせる。

---

## 4. スタンプ付与API（LIFF側で呼んでいる API）の役割

- **入力:** LINEユーザーID（LIFFから渡す）、**`stamps`（付与するスタンプ個数）**、任意で `qr_code_id`。
- **処理:**
  1. `profiles` から `line_user_id`（または `id`）で該当患者を取得。
  2. 該当がなければ 404 または 400。
  3. **QRの `stamps` の値に応じて**、スタンプを加算する（1回のスキャンで 優良=10 または 通常=5 スタンプ）。`stamp_history` へのINSERTや `profiles.stamp_count` の更新は、既存のスタンプ登録APIの仕様に合わせる。**ポイントではなくスタンプ個数で扱うこと。**
  4. 既存のDBトリガーで `profiles.stamp_count` 等が更新される。
- **重複防止:** 同一 `qr_code_id` + 同一 `user_id` で既に登録済みの場合は、409 Conflict または 200（冪等）で返すなどの方針を決める。

※ **`POST /api/stamps` を呼んでいる場合、リクエストBodyに「今回付与するスタンプ数」を渡す仕様にし、QRで読み取った `stamps` をそのまま渡すこと。** 固定で「1スタンプ」だけ加算していると、通常患者様用QR（5スタンプ）を読んでも 1 しか付かない。

## 5. テスト手順（想定）

1. 管理ダッシュボードで「テストQR」タブを開き、2種類のQRを表示する。
2. テスト端末でLIFFアプリを開き、カメラで「優良患者様用」のQRを読み取る。
3. LIFF側でAPIを呼び、成功したら「10スタンプ付与されました」等を表示。
4. 管理画面の患者一覧またはスタンプ履歴で、該当患者のスタンプが増えていることを確認する。
5. 同様に「通常患者様用」のQRで 5スタンプ付与 を確認する。

---

## 6. 本番用QRとの違い（将来拡張）

- **テスト用:** 固定の `{"type":"premium","stamps":10}` / `{"type":"regular","stamps":5}` のみ。`qr_code_id` は省略または簡易値で可。
- **本番用（案）:**  
  - 院内QRには、重複防止のため「1回限り有効なトークン」や「日付+QR設置場所ID」などを含め、サーバーで検証する方式を検討する。  
  - その場合、ペイロード形式の拡張（例: `token`, `expires_at`）と、API側の検証仕様を別途定義する。
