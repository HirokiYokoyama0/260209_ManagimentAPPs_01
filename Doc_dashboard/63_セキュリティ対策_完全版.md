### 根本原因

**RLSポリシーが `USING (true)` で実装されている**

```sql
-- 危険なポリシーの例
CREATE POLICY "allow_public_read" ON profiles
  FOR SELECT
  TO anon, authenticated
  USING (true);  -- ⚠️ 誰でもアクセス可能
```

このポリシーは以下を許可：
- ❌ ANON_KEYで全データにアクセス
- ❌ `.eq()` フィルタを削除すれば全件取得
- ❌ 他人のデータの閲覧・変更

## 実装内容

### 実装方針

**形式チェック方式**を採用：
- RLSポリシーに正規表現パターンを追加
- 既存のクライアントコードは変更不要
- 全件取得攻撃のみをブロック

### 対象テーブル（8テーブル）

1. profiles
2. stamp_history
3. reward_exchanges
4. families
5. patient_dental_records
6. milestone_history
7. event_logs
8. families_parent_permissions

### 実装パターン

```sql
-- 修正前（危険）
CREATE POLICY "allow_public_read" ON profiles FOR SELECT
  USING (true);  -- ⚠️ 誰でもアクセス可能

-- 修正後（安全）
CREATE POLICY "profiles_read_with_format_check" ON profiles FOR SELECT
  TO anon, authenticated
  USING (
    -- 本番LINE User ID（U + 32文字の16進数）
    id ~ '^U[0-9a-f]{32}$' OR
    -- テストLINE User ID（U_test_で始まる）
    id ~ '^U_test_' OR
    -- 代理管理メンバー（manual-child-で始まる）
    id LIKE 'manual-child-%'
  );
```

### 既存機能への影響

**クライアント側コード変更: ゼロ ✅**

理由:
- 全てのクエリが既に `.eq("id", userId)` を使用
- RLSポリシーは形式チェックのみ追加
- `.eq()` があれば通常通り動作

```typescript
// 既存コード（変更不要）
const { data } = await supabase
  .from("profiles")
  .select("*")
  .eq("id", userId)  // ← これがあれば動作する
  .single();
```

### セキュリティ効果

| 攻撃タイプ | Before | After |
|-----------|--------|-------|
| 全件取得攻撃 | ❌ 可能 | ✅ 防御 |
| ランダムスキャン攻撃 | ❌ 可能 | ✅ 防御 |
| 正規IDが既知の場合 | ❌ 可能 | ⚠️ 依然可能 |
| **総合評価** | **⭐ 最低** | **⭐⭐⭐ 中** |

**防げる攻撃:**
```javascript
// ❌ 全件取得しようとすると...
const { data } = await supabase.from("profiles").select("*");
// → RLSポリシーで弾かれる ✅

// ❌ 不正な形式のIDでアクセスしようとすると...
const { data } = await supabase
  .from("profiles")
  .select("*")
  .eq("id", "invalid-id");
// → 形式チェックで弾かれる ✅
```

**防げない攻撃（今後の課題）:**
```javascript
// ⚠️ 正規のLINE User IDが既知の場合
const { data } = await supabase
  .from("profiles")
  .select("*")
  .eq("id", "U5c70cd61f4fe89a65381cd7becee8de3");
// → 形式が正しいのでアクセスできる ⚠️
```

### トラブルシューティング

#### 問題1: LINEミニアプリでエラーが発生

**症状:**
```
Error: new row violates row-level security policy
```

**原因:**
- LINE User IDの形式が想定外
- テストユーザーのパターンが異なる

**対処:**
1. エラーログから実際のLINE User IDを確認
2. `check-line-user-id-length.mjs` でパターンを再確認
3. 必要に応じてRLSポリシーのパターンを追加

#### 問題2: テストユーザーがアクセスできない

**症状:**
```
テストユーザーでログインすると「データが取得できません」
```

**原因:**
- `U_test_` パターンが追加されていない

**対処:**
```sql
-- パターンを確認
SELECT policyname, qual
FROM pg_policies
WHERE tablename = 'profiles'
  AND policyname LIKE '%read%';

-- U_test_ が含まれていない場合は追加
```

#### 問題3: 管理ダッシュボードでエラーが発生

**症状:**
```
管理ダッシュボードで患者一覧が表示されない
```

**原因:**
- SERVICE_ROLE_KEY が設定されていない可能性

**対処:**
```bash
# 環境変数を確認
cat .env.local | grep SUPABASE_SERVICE_ROLE_KEY

# 設定されていない場合は追加
echo "SUPABASE_SERVICE_ROLE_KEY=your_service_role_key" >> .env.local
```
