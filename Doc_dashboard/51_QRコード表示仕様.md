## Part A: LIFF側 読み取り・API連携仕様

### お詫び（管理側）

過去に「優良患者様用＝1スタンプ/回」と誤って記載していたことがあり、LIFFアプリ側の開発・検証にご迷惑をおかけしました。申し訳ありません。
**正しくは 優良＝10スタンプ/回・通常＝5スタンプ/回 です。** 本ドキュメントおよび管理ダッシュボードのQRペイロードはこの仕様に統一済みです。

### よくある質問

**Q. ダッシュボード側のQRコード生成コードはどこにありますか？**
**A.** 管理ダッシュボード（本リポジトリ）の **`app/admin/qr/page.tsx`** です。`QRCodeSVG` で2種類のQRを表示しています。

**Q. QRコードのペイロード（中身のJSON）は何ですか？**
**A.** 次の2種類です（**スタンプ個数に統一。ポイントは廃止**）。

| 種類 | ペイロード文字列 |
|------|------------------|
| 優良患者様用 | `{"type":"premium","stamps":10}` |
| 通常患者様用 | `{"type":"regular","stamps":5}` |

**Q. `POST /api/stamps` が呼ばれているが、通常患者用QRでも1スタンプしか付かない**
**A.** ダッシュボードのQRは **stamps:10（優良）/ stamps:5（通常）** を出しています。**APIのリクエストBodyに、QRの `stamps` をそのまま渡すよう実装を確認してください。**

---

### 1. 概要

- 患者が**来院時**に、院内に設置されたQRコード（またはテスト用として管理画面に表示されたQR）を、**LIFFアプリ内のカメラで読み取る**。
- 読み取った内容に応じて、**スタンプ付与API** を呼び出し、`stamp_history` に1件登録する。これにより `profiles.stamp_count` 等が既存トリガーで自動更新される。

---

### 2. QRコードのデータ形式（ペイロード）

**※ ポイント概念は廃止済み。スタンプは「個数」に統一。**

#### 2.1 優良患者様用（10スタンプ/回）

```json
{"type":"premium","stamps":10}
```

#### 2.2 通常患者様用（5スタンプ/回）

```json
{"type":"regular","stamps":5}
```

#### 2.3 共通

- 将来的に本番用QRで `qr_code_id`（重複防止用）や `expires_at` などを含める場合は、LIFF側でそれらをパースし、APIに渡す想定。
- **旧仕様の `points` は使用しないこと。** LIFF側は必ず **stamps** を参照し、APIには付与する**スタンプ個数**を渡すこと。

---

### 3. LIFF側の処理フロー

1. **カメラ起動**: LIFFアプリ内でカメラを起動し、QRコードを読み取る（LIFF SDK または HTML5 の `getUserMedia` + 任意のQRデコードライブラリ）。

2. **デコード**: 読み取った文字列が上記JSON形式であればパースする。**`type` と `stamps`（スタンプ個数）を取得。** 不正な形式の場合はエラー表示し、再スキャンさせる。

3. **スタンプ付与APIの呼び出し**:
   - **エンドポイント:** `POST /api/stamps` に **QRの `stamps` の値を渡す**。
   - **リクエスト例:**
     ```json
     {
       "type": "premium",
       "stamps": 10,
       "qr_code_id": "optional-unique-id-for-dedup"
     }
     ```
   - **認証:** LIFFの `getProfile()` で取得した `userId` をヘッダーまたはBodyに含める。
   - **重複防止:** `qr_code_id` をリクエストに含め、サーバー側で「同一 `qr_code_id` + 同一ユーザー」の連続登録を制限する設計が望ましい。

4. **レスポンス**:
   - 成功: 200 + 付与後のスタンプ数など
   - 失敗: 4xx/5xx + エラー内容

---

### 4. スタンプ付与APIの役割

- **入力:** LINEユーザーID、**`stamps`（付与するスタンプ個数）**、任意で `qr_code_id`
- **処理:**
  1. `profiles` から該当患者を取得
  2. `stamp_history` へINSERT → DBトリガーで `profiles.stamp_count` が自動更新
- **重複防止:** 同一 `qr_code_id` + 同一 `user_id` で既に登録済みの場合は 409 Conflict

## Part B: テスト用QRコード表示（管理ダッシュボード側）

### 1. 概要

- 開発・テスト時に実物QRを用意しにくいため、**管理ダッシュボードに「テスト用QRコード」を2つ表示するタブ**を設ける。
- スタッフが画面に表示したQRを、テスト端末のLIFFアプリで読み取り、スタンプ付与フローを確認できる。

### 2. 画面・ルート

- **パス:** `/admin/qr`
- **タブ名:** 「テストQR」
- **実装:** `app/admin/qr/page.tsx`

### 3. 画面レイアウト

- タイトル: 「テスト用QRコード」
- 2つのブロックを横並び（PC）／縦並び（スマホ）:
  - 各ブロックに「ラベル」「QRコード画像」「付与スタンプ数（10／5）」「ペイロード確認用文字列」を表示
- QRコード生成: `qrcode` npm パッケージで SVG を生成して表示

### 4. テスト手順

1. 管理ダッシュボードで「テストQR」タブを開き、2種類のQRを表示
2. テスト端末でLIFFアプリを開き、カメラで「優良患者様用」のQRを読み取る
3. LIFF側でAPIを呼び、成功したら「10スタンプ付与されました」等を表示
4. 管理画面の患者一覧で該当患者のスタンプが増えていることを確認
5. 同様に「通常患者様用」のQRで 5スタンプ付与 を確認

---

## Part C: 本番用QRとの違い（将来拡張）

- **テスト用:** 固定ペイロードのみ。`qr_code_id` は省略可。
- **本番用（案）:** 院内QRには「1回限り有効なトークン」や「日付+QR設置場所ID」などを含め、サーバーで検証する方式を検討する。
