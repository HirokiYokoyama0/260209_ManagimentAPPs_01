### システム構成

```
┌─────────────────────────────────────────────┐
│          LINEミニアプリ（LIFF）              │
│  - 患者向けUIアプリ                         │
│  - LINE認証（LINE User ID）                │
│  - Supabase ANON_KEY使用                   │
└────────────────┬────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────┐
│           Supabase PostgreSQL               │
│  - RLS (Row Level Security) 有効           │
│  - データベース本体                         │
└────────────────┬────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────┐
│        管理ダッシュボード（Next.js）         │
│  - スタッフ向けWebアプリ                    │
```

- **開発環境**: ローカル開発中（本番デプロイ未実施）
- **データベース**: Supabase 無料プラン (500MB)
- **個人情報**: 含まれる（氏名、診察券番号、LINE User ID、歯科記録）
- **法的要件**: 個人情報保護法、医療情報保護ガイドライン

## 脅威モデル

### 想定する脅威

#### 1. 外部からの脅威

| 脅威 | リスクレベル | 対策状況 |
|-----|------------|---------|
| 不正アクセス（管理画面） | 🔴 高 | ⚠️ 簡易認証のみ |
| SQLインジェクション | 🟡 中 | ✅ SupabaseのORM使用 |
| XSS（クロスサイトスクリプティング） | 🟡 中 | ✅ Reactの自動エスケープ |
| CSRF（クロスサイトリクエストフォージェリ） | 🟡 中 | ✅ SameSite Cookie |
| 総当たり攻撃（パスワード） | 🔴 高 | ❌ レート制限なし |
| DDoS攻撃 | 🟢 低 | △ Vercel/Supabaseの保護に依存 |

#### 2. 内部からの脅威

| 脅威 | リスクレベル | 対策状況 |
|-----|------------|---------|
| スタッフによる不正操作 | 🟡 中 | ✅ activity_logs で記録 |
| 機密情報の漏洩 | 🔴 高 | ⚠️ .env.local の管理に依存 |
| 誤操作によるデータ削除 | 🟡 中 | △ バックアップなし |

#### 3. アプリケーション固有の脅威

| 脅威 | リスクレベル | 対策状況 |
|-----|------------|---------|
| LINE User ID の偽装 | 🟡 中 | ✅ LIFF SDKによる検証 |
| ANON_KEY の流出 | 🟡 中 | ✅ RLSで保護 |
| SERVICE_ROLE_KEY の流出 | 🔴 高 | ⚠️ .env.local のみ |

## セキュリティ設計原則

### 1. 多層防御（Defense in Depth）

複数の防御層を設けることでリスクを低減：

```
┌────────────────────────────────────┐
│ Layer 1: ネットワーク              │  HTTPS強制
├────────────────────────────────────┤
│ Layer 2: 認証・認可                │  スタッフ認証、LINE認証
├────────────────────────────────────┤
│ Layer 3: アプリケーション          │  入力検証、エスケープ
├────────────────────────────────────┤
│ Layer 4: データベース              │  RLS、暗号化
├────────────────────────────────────┤
│ Layer 5: 監査・検知                │  activity_logs, event_logs
└────────────────────────────────────┘
```

### 2. 最小権限の原則（Principle of Least Privilege）

**現状の問題点:**
- profiles, families, reward_exchanges などのテーブルで `USING (true)` ポリシー
- すべてのユーザーが全データにアクセス可能

**改善方針:**
- ユーザーは自分のデータのみアクセス可能にする
- スタッフは管理画面経由でのみ操作可能にする

### 3. 信頼の分離（Trust Boundaries）

| システム | 信頼レベル | キー種別 | 権限範囲 |
|---------|-----------|---------|---------|
| LIFFアプリ | 不特定多数 | ANON_KEY | RLSで制限 |
| 管理ダッシュボード | スタッフのみ | SERVICE_ROLE_KEY | RLSバイパス |

### 4. セキュアバイデフォルト

- RLSは常に有効化
- デフォルトで拒否、明示的に許可
- 開発環境でも本番に近いセキュリティ設定を維持

## データ分類と保護方針

### データ分類

| 分類 | 例 | 保護レベル | 暗号化 | バックアップ |
|-----|---|-----------|-------|------------|
| **機密情報** | パスワードハッシュ、SERVICE_ROLE_KEY | 🔴 最高 | 必須 | 必須 |
| **個人情報** | 本名、診察券番号、LINE User ID | 🟠 高 | 推奨 | 必須 |
| **医療情報** | 歯科ケア記録（tooth_data） | 🟠 高 | 推奨 | 必須 |
| **準個人情報** | LINE表示名、プロフィール画像URL | 🟡 中 | 任意 | 必須 |
| **統計情報** | スタンプ数、来院回数 | 🟢 低 | 任意 | 推奨 |
| **ログ情報** | activity_logs, event_logs | 🟡 中 | 任意 | 必須 |

### 個人情報を含むテーブル

```sql
profiles
├── real_name (本名) ⚠️ 個人情報
├── ticket_number (診察券番号) ⚠️ 識別子
├── line_user_id (LINE User ID) ⚠️ 識別子
├── display_name (LINE表示名) △ 準個人情報
└── picture_url (プロフィール画像) △ 準個人情報

patient_dental_records
├── patient_id (患者ID) ⚠️ 識別子
├── tooth_data (歯の状態) ⚠️ 医療情報
└── staff_memo (スタッフメモ) ⚠️ 医療情報

staff
├── login_id (ログインID) ⚠️ 識別子
└── password_hash (パスワードハッシュ) 🔴 機密情報
```

## アクセス制御方針

### 管理ダッシュボード側

**現在の実装:**
```typescript
// 認証方式: 独自実装（簡易認証）
// - staff テーブル（bcrypt ハッシュ）
// - 環境変数フォールバック（ADMIN_USER / ADMIN_PASSWORD）
// - セッション管理: HMAC署名付きCookie（7日間有効）
// - デバイス別セッション: admin_session_{deviceId}
```

**セキュリティ上の懸念:**
1. ❌ **レート制限なし** → 総当たり攻撃に脆弱
2. ❌ **多要素認証なし** → アカウント乗っ取りリスク
3. ⚠️ **環境変数に平文パスワード** → .env.local漏洩リスク
4. ⚠️ **セッション有効期限7日** → 長すぎる可能性

### LIFFアプリ側

**現在の実装:**
```typescript
// 認証方式: LINE LIFF SDK
// - LINE User ID による識別
// - liff.init() → liff.getProfile()
// - Supabase ANON_KEY 使用
```

**セキュリティ上の懸念:**
1. ✅ LINE SDKが認証を保証
2. ⚠️ **RLSポリシーが緩い** → 他人のデータにアクセス可能
3. ⚠️ **ANON_KEYがクライアント側に露出** → 必然的だがRLSで防御が必須

## 監査・ロギング方針

### ログの種類

#### 1. スタッフ操作ログ（activity_logs）

**目的:** 監査、不正操作の検知

| 記録内容 | 保存期間 | アクセス権限 |
|---------|---------|------------|
| 操作者（staff_id） | 無期限 | 管理者のみ |
| 操作種別（action） | 無期限 | 管理者のみ |
| 対象（target_type, target_id） | 無期限 | 管理者のみ |
| 詳細（details: JSONB） | 無期限 | 管理者のみ |
| 日時（created_at） | 無期限 | 管理者のみ |

**記録される操作:**
- login, logout
- profile_update, stamp_increment, stamp_set
- message_send, broadcast_send
- reward_exchange_complete, reward_exchange_cancel, reward_exchange_delete
- family_create, family_update, family_delete
- staff_create, staff_update, staff_deactivate
- survey_targets_distribute, survey_answer_reset

**RLS設定:**
```sql
ALTER TABLE activity_logs ENABLE ROW LEVEL SECURITY;
-- ポリシーなし = SERVICE_ROLE_KEYのみアクセス可能
```

#### 2. ユーザー行動ログ（event_logs）

**目的:** ユーザー行動分析、マーケティング

| 記録内容 | 保存期間 | アクセス権限 |
|---------|---------|------------|
| ユーザーID（user_id） | 400日 | 管理者のみ |
| イベント名（event_name） | 400日 | 管理者のみ |
| 流入元（source） | 400日 | 管理者のみ |
| メタデータ（metadata: JSONB） | 400日 | 管理者のみ |

**記録されるイベント:**
- app_open, page_view
- stamp_scan_success, stamp_scan_fail
- reward_exchange_success
- reservation_button_click
- slot_game_play, slot_game_win

## インシデント対応方針

### インシデント分類

| レベル | 例 | 初動対応時間 | 対応者 |
|-------|---|-------------|--------|
| 🔴 Critical | SERVICE_ROLE_KEY漏洩、データベース全削除 | 即座 | 開発者 |
| 🟠 High | 不正ログイン、個人情報漏洩 | 1時間以内 | 開発者 |
| 🟡 Medium | ログ異常、アクセス集中 | 4時間以内 | 開発者 |
| 🟢 Low | 軽微なエラー | 1営業日以内 | 開発者 |

### インシデント対応フロー

```
1. 検知 → 2. 初動対応 → 3. 調査 → 4. 恒久対策 → 5. 報告・記録
```

#### 1. SERVICE_ROLE_KEY 漏洩時の対応

```bash
# 即座に実施
1. Supabase ダッシュボードで SERVICE_ROLE_KEY を再生成
2. .env.local を更新
3. アプリケーションを再起動
4. activity_logs で不審な操作がないか確認
5. 必要に応じてデータをロールバック
```

#### 2. 不正ログイン検知時の対応

```bash
1. 該当スタッフのアカウントを無効化（is_active = false）
2. セッションCookieを無効化
3. パスワードリセットを強制
4. activity_logs で操作履歴を確認
5. 影響範囲を特定し、必要に応じてデータを修正
```

#### 🔴 Critical（必須）

- [ ] **1. .env.local の本番用設定**
  ```bash
  # ✅ 実施項目
  - ADMIN_PASSWORD を強力なランダム文字列に変更
  - AUTH_SECRET を長いランダム文字列に変更（最低32文字）
  - .env.local を .gitignore に確実に含める
  ```

- [ ] **2. RLSポリシーの厳格化**
  ```sql
  -- profiles テーブル
  DROP POLICY "allow_public_read" ON profiles;
  DROP POLICY "allow_public_update" ON profiles;

  CREATE POLICY "user_read_own_profile"
    ON profiles FOR SELECT
    USING (auth.uid() = id OR id = ANY(SELECT id FROM profiles));
    -- ⚠️ 注: Supabase Authを使用していないため、別の方法を検討

  -- reward_exchanges テーブル
  DROP POLICY "allow_public_read_exchanges" ON reward_exchanges;

  CREATE POLICY "user_read_own_exchanges"
    ON reward_exchanges FOR SELECT
    USING (user_id = current_user_id());
  ```
