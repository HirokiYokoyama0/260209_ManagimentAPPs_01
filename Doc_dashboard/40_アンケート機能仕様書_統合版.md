## 3. データベース設計

### 3.1 surveys テーブル

**アンケート定義マスター**

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | TEXT | NO | アンケートID（主キー、例: 'satisfaction_2026Q1'） |
| `title` | TEXT | NO | アンケートタイトル |
| `description` | TEXT | YES | 説明文 |
| `reward_stamps` | INTEGER | NO | 報酬スタンプ数（個・整数） |
| `is_active` | BOOLEAN | NO | 公開中かどうか |
| `start_date` | TIMESTAMPTZ | YES | 公開開始日 |
| `end_date` | TIMESTAMPTZ | YES | 公開終了日 |
| `created_at` | TIMESTAMPTZ | NO | 作成日時 |

**インデックス:**
- PRIMARY KEY: `id`

---

### 3.2 survey_answers テーブル

**回答データ（Phase 1: 固定質問スキーマ）**

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | UUID | NO | 主キー |
| `user_id` | TEXT | NO | 回答者（profiles.id） |
| `survey_id` | TEXT | NO | アンケートID（surveys.id） |
| `q1_rating` | INTEGER | YES | Q1: 5段階評価（1〜5） |
| `q2_comment` | TEXT | YES | Q2: 自由記述 |
| `q3_recommend` | INTEGER | YES | Q3: 推奨度（0〜10、NPS計算用） |
| `created_at` | TIMESTAMPTZ | NO | 回答日時 |

**制約:**
- UNIQUE: `(user_id, survey_id)` - 重複回答防止

**インデックス:**
- `idx_survey_answers_survey_id` ON `survey_id`
- `idx_survey_answers_user_id` ON `user_id`
- `idx_survey_answers_created_at` ON `created_at`

---

### 3.3 survey_targets テーブル

**配信対象者管理**

| カラム名 | 型 | NULL | 説明 |
|---------|-----|------|------|
| `id` | UUID | NO | 主キー |
| `user_id` | TEXT | NO | 対象ユーザー（profiles.id） |
| `survey_id` | TEXT | NO | アンケートID（surveys.id） |
| `show_on_liff_open` | BOOLEAN | NO | LIFFアプリ初期表示するか |
| `answered_at` | TIMESTAMPTZ | YES | 回答完了日時（NULLなら未回答） |
| `postponed_count` | INTEGER | NO | 「あとで」を押した回数 |
| `last_postponed_at` | TIMESTAMPTZ | YES | 最後に「あとで」を押した日時 |
| `created_at` | TIMESTAMPTZ | NO | 配信設定日時 |
| `updated_at` | TIMESTAMPTZ | NO | 更新日時 |

**制約:**
- UNIQUE: `(user_id, survey_id)` - 重複配信防止

**インデックス:**
- `idx_survey_targets_user_id` ON `user_id`
- `idx_survey_targets_survey_id` ON `survey_id`
- `idx_survey_targets_show_on_liff` ON `show_on_liff_open` WHERE `show_on_liff_open = true`
- `idx_survey_targets_answered` ON `answered_at`

### 3.4 survey_target_status ビュー

**管理画面用：配信対象者の一覧**

```sql
create or replace view survey_target_status as
select
  st.id,
  st.survey_id,
  s.title as survey_title,
  st.user_id,
  p.real_name,
  p.display_name,
  p.ticket_number,
  st.show_on_liff_open,
  st.answered_at,
  case
    when st.answered_at is not null then '回答済み'
    else '未回答'
  end as status,
  st.postponed_count,
  st.last_postponed_at,
  st.created_at
from survey_targets st
join surveys s on st.survey_id = s.id
join profiles p on st.user_id = p.id;
```

**注意:** PostgreSQLの仕様上、VIEW内のORDER BYは結果順序を保証しません。
API実装時に必ず `.order()` を明示的に指定してください。

### 3.5 increment_survey_postponed RPC関数

**「あとで」ボタンのカウント増加**

```sql
create or replace function increment_survey_postponed(
  p_user_id text,
  p_survey_id text
)
returns void as $$
begin
  update survey_targets
  set
    postponed_count = postponed_count + 1,
    last_postponed_at = now(),
    updated_at = now()
  where user_id = p_user_id
    and survey_id = p_survey_id;
end;
$$ language plpgsql security definer;
```

### 3.6 RLSポリシー

**surveys テーブル:**
```sql
-- 公開中のアンケートは全員が参照可能
create policy "anon_can_read_active_surveys"
  on surveys for select
  using (is_active = true);
```

**survey_answers テーブル:**
```sql
-- anon keyで全件参照・挿入可能（アプリ層でuser_idフィルタ）
create policy "anon_can_read_own_answers"
  on survey_answers for select
  using (true);

create policy "anon_can_insert_own_answers"
  on survey_answers for insert
  with check (true);
```

**survey_targets テーブル:**
```sql
-- anon keyで全件参照・更新可能（アプリ層でuser_idフィルタ）
create policy "anon_can_read_own_targets"
  on survey_targets for select
  using (true);

create policy "anon_can_update_own_targets"
  on survey_targets for update
  using (true);
```

**注意:** 管理者（service_role）は全件操作可能（RLSをバイパス）

### 3.7 スタンプ報酬の付与

アンケート回答時のスタンプ付与は、既存のスタンプシステムに統合されています。

**stamp_history への記録:**
```typescript
await supabase.from('stamp_history').insert({
  user_id: profile.id,
  visit_date: new Date().toISOString(),
  stamp_number: profile.stamp_count + (reward_stamps * 10),
  amount: reward_stamps * 10,  // 3個 = 30（内部は10倍）
  stamp_method: 'survey_reward',  // アンケート報酬専用
  notes: `アンケート回答: ${surveyId}`,
});
```

**stamp_method の値:**
- `'qr_scan'` - QRコードスキャン
- `'manual_admin'` - 管理者が手動付与
- `'import'` - インポート
- **`'survey_reward'`** - アンケート回答報酬 ⭐

**注意:** トリガー `update_profile_stamp_count` が自動的に `profiles.stamp_count` を更新します

## 6. LIFFアプリ側の仕様

### 6.1 アプリ起動時の処理

LIFFアプリ起動時に、以下のロジックで未回答アンケートをチェックします。

```typescript
useEffect(() => {
  const checkPendingSurvey = async () => {
    const { data: targets } = await supabase
      .from('survey_targets')
      .select('survey_id, surveys(*)')
      .eq('user_id', profile.id)
      .eq('show_on_liff_open', true)
      .is('answered_at', null)
      .eq('surveys.is_active', true)
      .lt('postponed_count', 3);  // 3回「あとで」を押したら表示しない

    if (targets && targets.length > 0) {
      setSurveyModal({
        isOpen: true,
        surveyId: targets[0].survey_id,
        title: targets[0].surveys.title,
      });
    }
  };

  if (profile?.id) {
    checkPendingSurvey();
  }
}, [profile?.id]);
```

**表示パターン:** モーダル表示（推奨）

**操作:**
- 「今すぐ回答」 → アンケート画面へ遷移
- 「あとで」 → `increment_survey_postponed` RPC を呼び出し、モーダルを閉じる

### 6.3 「あとで」ボタンの挙動

```typescript
const handleSurveyLater = async () => {

    p_user_id: profile.id,

  });

```

**運用ルール:**
- 「あとで」を3回押したら、それ以降は表示しない
- `postponed_count < 3` の条件でフィルタ

## 7. 運用ガイド

### 7.1 アンケート配信の基本フロー

```
1. アンケート作成（タイトル・説明・報酬を設定）
   ↓
2. 配信対象者選択（全員/条件/個別）
   ↓
3. LIFF初期表示をONに設定
   ↓
4. 配信開始（survey_targets にレコード挿入）
   ↓
5. 回答状況のモニタリング
   ↓
6. 結果集計・CSV出力
```

### 7.2 配信対象者の選択方法

#### ① 全員配信
全登録ユーザーに配信します。

**使用例:** 定期的な満足度調査

#### ② 条件フィルタ配信

| フィルタ | 条件例 | 用途 |
|---------|-------|------|
| **最終来院日** | 30日以内 | 最近来院したユーザー |
| **最終来院日** | 90日以上前 | 休眠ユーザー |
| **スタンプ数** | 10個以上 | ヘビーユーザー |
| **スタンプ数** | 1個以上 | 初回来院済みユーザー |

**使用例:**
- 新規ユーザーへの初回満足度調査（登録後7日以内）
- ヘビーユーザーへのロイヤリティ調査（スタンプ10個以上）
- 休眠ユーザーへの離脱理由調査（最終来院日が90日以上前）

#### ③ 個別選択配信
診察券番号で検索して個別に選択します。

**使用例:** 特定イベント参加者への感想調査

### 7.3 回答状況の確認

**配信対象者管理画面で確認できる情報:**
- 配信対象者数
- 回答済み数
- 未回答数
- 回答率（%）
- 個別の回答状況（回答済み/未回答）
- 後回し回数

**対応アクション:**
- 回答率が低い場合 → LIFF初期表示をONにする
- 誤って回答済みにした場合 → 「未回答に戻す」機能を使用

### 7.4 結果の活用

**Q1（満足度評価）の活用:**
- 評価が低い（1-2）の割合が高い場合 → サービス改善が必要
- 評価が高い（4-5）の割合が高い場合 → 強みとして訴求

**NPS（推奨度）の活用:**
- NPS > 50: 非常に良好
- NPS 0-50: 良好
- NPS < 0: 改善が必要

**Q2（自由記述）の活用:**
- 具体的な改善点の把握
- 患者の声を公式サイトに掲載（許可を得て）

### 7.5 よくある質問

**Q: アンケートを一度配信した後、追加で対象者を増やせますか？**
A: はい。配信対象者管理画面で再度対象者を追加できます。

**Q: 回答を誤って送信したユーザーがいた場合、どうすれば？**
A: 「未回答に戻す」機能で回答をリセットできます。

**Q: LIFF初期表示をOFFにしたら、ユーザーはアンケートを見られなくなりますか？**
A: モーダル表示はされなくなりますが、LIFFアプリ内のアンケートページからは引き続きアクセス可能です。

**Q: 同じユーザーに複数のアンケートを配信できますか？**
A: はい。複数のアンケートを同時に配信できます。LIFF起動時は最初の1つがモーダル表示されます。
