## ❌ 問題：管理ダッシュボードが交換履歴を認識できない

### 症状
- ミニアプリ側で特典交換が成功している
- `reward_exchanges` テーブルにデータが入っている
- しかし管理ダッシュボードでは「交換履歴がありません」と表示される

### 推定原因
管理ダッシュボードが以下のいずれかを行っている可能性：

1. ❌ `rewards` テーブルと JOIN しようとしている
   - → 外部キー制約が削除されたため JOIN できない

2. ❌ `is_milestone_based = false` で絞り込んでいる
   - → 新しい特典はすべて `is_milestone_based = true`

3. ❌ 古いカラムのみを SELECT している
   - → `milestone_reached`, `is_milestone_based` などの新カラムが必要

---

## ✅ 正しいクエリ方法

### ⚠️ 重要：外部キー制約が削除されている

`reward_exchanges.reward_id` の外部キー制約（`reward_exchanges_reward_id_fkey`）は削除されています。

**理由**：`reward_id` が `rewards` または `milestone_rewards` の両方を参照できるようにするため。

**影響**：Supabaseの自動JOIN（`.select('*, reward:milestone_rewards(*)')`）は動作しません。

**対応**：手動でJOINする必要があります（後述）。

### 最小限のテーブル構成（3つだけ）

#### 1. **milestone_rewards** テーブル（特典マスター）
```sql
-- 特典の種類を定義
SELECT * FROM milestone_rewards WHERE is_active = true;
```

| id | name | milestone_type | reward_type |
|----|------|----------------|-------------|
| UUID | 歯ブラシ 1本 | every_10 | toothbrush |
| UUID | POIC殺菌剤 | every_50 | poic |
| UUID | 選べるメニュー割引 | every_150_from_300 | premium_menu |

#### 2. **reward_exchanges** テーブル（交換履歴）
```sql
-- 交換履歴を取得（マイルストーン型のみ）
SELECT
  re.*,
  mr.name AS reward_name,
  mr.milestone_type,
  mr.reward_type
FROM reward_exchanges re
LEFT JOIN milestone_rewards mr ON re.reward_id = mr.id
WHERE re.is_milestone_based = true
ORDER BY re.exchanged_at DESC;
```

**重要な新カラム：**
- `milestone_reached`: 何個目のスタンプで交換したか（10, 20, 50など）
- `is_milestone_based`: 必ず `true`（新仕様）
- `is_first_time`: POIC の初回判定用
- `valid_until`: 有効期限

#### 3. **profiles** テーブル（ユーザー情報）
```sql
SELECT id, stamp_count FROM profiles;
```

## 🔧 必要な修正

### 1. JOIN先の変更

**❌ 古い（動かない）：**
```sql
SELECT re.*, r.name
FROM reward_exchanges re
JOIN rewards r ON re.reward_id = r.id;  -- rewards テーブルは空！
```

**✅ 新しい（正しい）：**

**方法1: SQL（推奨）**
```sql
SELECT re.*, mr.name
FROM reward_exchanges re
LEFT JOIN milestone_rewards mr ON re.reward_id = mr.id
WHERE re.is_milestone_based = true;
```

**方法2: アプリケーション側で手動JOIN（外部キー制約がないため）**
```javascript
// Step 1: reward_exchanges を取得
const { data: exchanges } = await supabase
  .from('reward_exchanges')
  .select('*')
  .eq('is_milestone_based', true)
  .order('exchanged_at', { ascending: false });

// Step 2: milestone_rewards を取得
const { data: rewards } = await supabase
  .from('milestone_rewards')
  .select('*');

// Step 3: 手動でJOIN
const exchangesWithRewards = exchanges.map(ex => {
  const reward = rewards.find(r => r.id === ex.reward_id);
  return {
    ...ex,
    reward_name: reward?.name || '不明',
    milestone_type: reward?.milestone_type,
    reward_type: reward?.reward_type
  };
});
```

**注意**：Supabaseの自動JOIN（`.select('*, reward:milestone_rewards(*)')`）は、外部キー制約が削除されているため動作しません。

### 2. 特典名の表示

**reward_id から特典名を取得する場合：**

```sql
-- milestone_rewards を参照
SELECT name FROM milestone_rewards WHERE id = 'reward_id';
```

### 3. マイルストーンの表示

交換履歴に「何個目で交換したか」を表示する場合：

```sql
SELECT
  re.milestone_reached,  -- 10, 20, 50 など
  mr.name,
  re.status,
  re.exchanged_at
FROM reward_exchanges re
LEFT JOIN milestone_rewards mr ON re.reward_id = mr.id
WHERE re.is_milestone_based = true;
```

表示例：
- `10個目：歯ブラシ 1本（pending）`
- `50個目：POIC殺菌剤（pending）`

## 🚨 重要な変更点

### 1. 外部キー制約の削除
```sql
-- 実行済み
ALTER TABLE reward_exchanges
DROP CONSTRAINT IF EXISTS reward_exchanges_reward_id_fkey;
```

**理由**: `reward_id` が `rewards` または `milestone_rewards` の両方を参照できるようにするため

### 2. 重複チェックの変更

同じ特典（例：歯ブラシ）でも、**マイルストーンが違えば別の申請**として扱います。

**例：**
- 10個目の歯ブラシ（`milestone_reached: 10`）
- 20個目の歯ブラシ（`milestone_reached: 20`）
- 50個目の歯ブラシ（`milestone_reached: 50`）

→ これらは**すべて別の交換履歴**

### 3. テーブルの役割

| テーブル | 役割 | 参照方法 |
|---------|------|---------|
| **milestone_rewards** | 特典の種類（マスターデータ） | `SELECT * WHERE is_active = true` |
| **reward_exchanges** | 交換履歴（新旧両対応） | `WHERE is_milestone_based = true` で新仕様のみ |
| **profiles** | ユーザーのスタンプ数 | `stamp_count` カラム |

## 🔍 原因分析：なぜ管理ダッシュボードで交換履歴が表示されないのか

### 検証結果

以下のテストを実施しました（`scripts/test-dashboard-query.mjs`）。

#### ❌ ケース1: rewards テーブルと JOIN しようとした場合

```javascript
const { data, error } = await supabase
  .from('reward_exchanges')
  .select('*, reward:rewards(name, description)')
  .eq('is_milestone_based', true);
```

**結果**：
```
エラー: Could not find a relationship between 'reward_exchanges' and 'rewards' in the schema cache
```

**原因**：
- 外部キー制約（`reward_exchanges_reward_id_fkey`）が削除されている
- Supabaseは外部キー制約を使って自動JOINを行うため、制約がないと動作しない
- さらに、`reward_id` には `milestone_rewards.id` が格納されているため、`rewards` テーブルには存在しない

---

#### ❌ ケース2: is_milestone_based = false で絞り込んだ場合

```javascript
const { data } = await supabase
  .from('reward_exchanges')
  .select('*')
  .eq('is_milestone_based', false);
```

**結果**：`0 件`

**原因**：
- 新しい特典交換は `is_milestone_based = true` で記録される
- `false` または `NULL` で絞り込むと、新しいデータは取得できない

---

#### ❌ ケース3: milestone_reached を取得していない場合

```javascript
const { data } = await supabase
  .from('reward_exchanges')
  .select('id, user_id, reward_id, status')
  .limit(5);
```

**結果**：
```
1 件
- ID: dec354d3..., status: pending
```

**問題**：
- `milestone_reached` が取得されないため、「何スタンプ目の交換か」がわからない
- 表示が不完全になる

---

#### ✅ ケース4: 手動で JOIN した場合（正しい方法）

```javascript
// Step 1: reward_exchanges を取得
const { data: exchanges } = await supabase
  .from('reward_exchanges')
  .select('*')
  .eq('is_milestone_based', true)
  .order('exchanged_at', { ascending: false });

// Step 2: milestone_rewards を取得
const { data: rewards } = await supabase
  .from('milestone_rewards')
  .select('*');

// Step 3: 手動でJOIN
exchanges.forEach(ex => {
  const reward = rewards.find(r => r.id === ex.reward_id);
  console.log(`特典名: ${reward?.name}`);
  console.log(`milestone_reached: ${ex.milestone_reached}`);
});
```

**結果**：
```
reward_exchanges: 1 件
milestone_rewards: 3 件

- ID: dec354d3...
  特典名: 歯ブラシ 1本
  reward_type: toothbrush
  milestone_reached: 10
  status: pending
  exchanged_at: 2026-03-27T14:32:47.985308+00:00
```

**✅ 成功**：データが正しく取得できた

---

### 結論

管理ダッシュボードが交換履歴を認識できない原因は、以下のいずれか（または複数）です：

1. ❌ `rewards` テーブルと JOIN しようとしている
   - → 外部キー制約がないため失敗
   - → `milestone_rewards` テーブルを使う必要がある

2. ❌ `is_milestone_based = false` または `NULL` で絞り込んでいる
   - → 新しい特典は `is_milestone_based = true`
   - → `WHERE is_milestone_based = true` が必要

3. ❌ `milestone_reached` カラムを取得していない
   - → 何スタンプ目の交換かがわからない
   - → SELECT に含める必要がある

4. ❌ Supabaseの自動JOIN（`.select('*, reward:milestone_rewards(*)')`）を使おうとしている
   - → 外部キー制約がないため動作しない
   - → 手動JOINまたはSQLの LEFT JOIN が必要
