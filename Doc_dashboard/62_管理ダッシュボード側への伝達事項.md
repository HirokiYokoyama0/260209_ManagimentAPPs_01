# 管理ダッシュボード側への伝達事項 - セキュリティ対策の影響

**作成日:** 2026-04-03
**対象:** 管理ダッシュボード開発者
**関連:** [62B_セキュリティ対策_ミニマム実装案.md](62B_セキュリティ対策_ミニマム実装案.md)

---

## 📋 概要

LINEミニアプリ側でセキュリティ対策を実施する際、管理ダッシュボード側にも影響が及ぶ可能性があります。このドキュメントでは、管理ダッシュボード側で確認・対応が必要な事項をまとめます。

---

## 🔐 現在のセキュリティ状況

### ✅ 管理ダッシュボード側（問題なし）

**現在の実装:**
```typescript
// 管理ダッシュボードは SERVICE_ROLE_KEY を使用
const supabase = createClient(
  supabaseUrl,
  SUPABASE_SERVICE_ROLE_KEY  // RLSをバイパス
);

// RLSポリシーの影響を受けない
const { data } = await supabase
  .from("profiles")
  .select("*");  // ✅ 全データにアクセス可能
```

**結論:**
- ✅ **管理ダッシュボード側は影響を受けません**
- SERVICE_ROLE_KEY は RLS をバイパスするため、RLSポリシー変更の影響なし
- 既存の実装をそのまま継続可能

### 🔴 LINEミニアプリ側（問題あり）

**現在の実装:**
```typescript
// ANON_KEY を使用
const supabase = createClient(
  supabaseUrl,
  SUPABASE_ANON_KEY  // RLSの影響を受ける
);

// RLSポリシー: USING (true) → 全データにアクセス可能
const { data } = await supabase
  .from("profiles")
  .select("*");  // ❌ 他人のデータも取得できてしまう
```

**問題:**
- RLSポリシーが `USING (true)` のため、誰でも全データにアクセス可能
- クライアント側のコードを改変すれば、他人のデータが閲覧・変更できる

## 📊 影響範囲の確認

### ✅ 影響を受けない項目（管理ダッシュボード）

| 項目 | 理由 |
|-----|------|
| **患者一覧の取得** | SERVICE_ROLE_KEY 使用のため ✅ |
| **患者詳細の閲覧** | SERVICE_ROLE_KEY 使用のため ✅ |
| **スタンプ数の変更** | SERVICE_ROLE_KEY 使用のため ✅ |
| **特典交換の管理** | SERVICE_ROLE_KEY 使用のため ✅ |
| **家族管理** | SERVICE_ROLE_KEY 使用のため ✅ |
| **スタッフ管理** | SERVICE_ROLE_KEY 使用のため ✅ |
| **ログの閲覧** | SERVICE_ROLE_KEY 使用のため ✅ |

### ⚠️ 確認が必要な項目

以下の項目について、管理ダッシュボード側で確認をお願いします:

#### 1. QRコード生成機能

**確認事項:**
- QRコード生成時のペイロード形式
- カメラQR vs ミニアプリQRの違い
- 15スタンプQRが正しく生成されているか

**背景:**
- [Doc 200](200_緊急バグ_15スタンプQRが10ポイントになる問題.md) で報告されている問題
- 15スタンプQRが10ポイントとして記録される事象が発生中

**依頼内容:**
```typescript
// QRコード生成ロジックの確認
// 優良患者QR（15スタンプ）のペイロード
{
  // カメラQR形式
  url: "https://liff.line.me/{liff-id}/auto-stamp?userId=...&amount=15&type=qr&location=premium"

  // ミニアプリQR形式
  payload: JSON.stringify({
    type: "premium",
    stamps: 15
  })
}
```

#### 2. Supabase接続の確認

**確認事項:**
- `.env` または `.env.local` に `SUPABASE_SERVICE_ROLE_KEY` が正しく設定されているか
- `SUPABASE_URL` が LINEミニアプリ側と同じか

**確認方法:**
```bash
# 管理ダッシュボードのルートディレクトリで実行
cat .env.local | grep SUPABASE

# 期待される出力
NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

#### 3. RPC関数の利用状況

**確認事項:**
- 管理ダッシュボードで以下のRPC関数を使用しているか？
  - `increment_reservation_clicks()`
  - `search_profiles_by_real_name()`
  - `get_latest_dental_record()`
  - その他カスタムRPC関数

**背景:**
- LINEミニアプリ側でRLSポリシーを変更すると、RPC関数の動作にも影響する可能性がある
- ただし、SERVICE_ROLE_KEY 使用時は影響を受けない

## 🛠️ セキュリティ対策の選択肢と影響

### 選択肢A: RLSポリシーのみ強化（推奨）

**LINEミニアプリ側の変更:**
```sql
-- RLSポリシーを厳格化
CREATE POLICY "users_can_read_own_profile"
  ON profiles FOR SELECT
  USING (line_user_id = current_line_user_id());
```

**管理ダッシュボード側の影響:**
- ✅ **影響なし**（SERVICE_ROLE_KEY がRLSをバイパスするため）

---

### 選択肢B: サーバーサイドAPI経由に変更

**LINEミニアプリ側の変更:**
```typescript
// Before: Supabase直接アクセス
const { data } = await supabase.from("profiles").select("*");

// After: API経由
const response = await fetch("/api/users/[userId]/profile");
const data = await response.json();
```

**管理ダッシュボード側の影響:**
- ✅ **影響なし**（管理ダッシュボードは引き続き Supabase 直接アクセスを継続）

---

### 選択肢C: Supabase Auth統合（非推奨 - 大規模変更）

**LINEミニアプリ側の変更:**
- LINE LIFF → Supabase Auth連携
- 認証フローの全面的な変更

**管理ダッシュボード側の影響:**
- ⚠️ **auth.users テーブルに LINE User ID が追加される**
- 既存の `profiles.line_user_id` との紐付けが必要
- ただし、SERVICE_ROLE_KEY 使用時は引き続きアクセス可能

---

## 🔍 確認していただきたいこと

### 1. QRコード生成ロジック（最優先）

**チェックポイント:**
```typescript
// ✅ 正しい実装例
const premiumQR = {
  type: "premium",
  stamps: 15,  // または amount: 15
};

const regularQR = {
  type: "regular",
  stamps: 10,  // または amount: 10
};

const purchaseQR = {
  type: "purchase",
  stamps: 5,  // または amount: 5
};
```

**確認方法:**
1. 管理ダッシュボードで15スタンプQRを生成
2. QRコードをスキャンしてペイロードを確認
3. `event_logs` テーブルで記録された `type` を確認

```sql
-- event_logs テーブルで確認
SELECT
  created_at,
  event_name,
  metadata->>'type' AS stamp_type,
  metadata->>'stamps_added' AS stamps_added,
  metadata->>'qr_raw_value' AS qr_raw_value
FROM event_logs
WHERE event_name IN ('stamp_scan_success', 'stamp_scan_success_in_app')
  AND created_at >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY created_at DESC
LIMIT 50;
```

### 2. 院内のQRコード実物確認

**チェックポイント:**
- 院内に複数のQRコードが設置されている場合、すべてが最新版か？
- 古いQRコード（10スタンプ）が混在していないか？

**確認方法:**
1. 院内のすべてのQRコードをスキャン
2. ペイロードを記録
3. 古いQRコードがあれば廃棄

### 3. SERVICE_ROLE_KEY の管理

**セキュリティチェック:**
```bash
# ❌ GitHub にコミットされていないか確認
cd /path/to/admin-dashboard
git log --all --full-history -- "*env*"

# ✅ .gitignore に含まれているか確認
cat .gitignore | grep env

# 期待される出力
.env
.env.local
.env*.local
```
