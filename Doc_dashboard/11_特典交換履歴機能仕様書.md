# 特典交換履歴機能仕様書

> 管理ダッシュボードに「特典交換履歴」タブを追加し、患者がスタンプを使って何を交換したかを把握できるようにする

**ステータス**: ✅ 実装完了（2026-02-14）
**最終更新**: 2026-03-28（マイルストーン型特典対応・スタッフ認証連携）

---

## 1. 概要

**目的**: 患者がLINEミニアプリ（LIFF）でマイルストーン到達時に獲得した特典の交換履歴を、管理ダッシュボードで一覧表示・管理する

**ユースケース**:
- 受付スタッフが「誰が何の特典を交換したか」を確認
- 特典の引き換え状況（pending/completed/cancelled）を管理
- 人気の特典を把握し、在庫管理や新しい特典企画の参考にする

**実装状況**: Phase 1完了

**マイルストーン型対応後の構造（2026-03-28更新）**:

| カラム名 | 型 | 説明 |
|---------|---|------|
| `id` | UUID | 主キー |
| `user_id` | TEXT | 交換した患者のID（profiles.id への参照） |
| `reward_id` | UUID | 交換した特典のID（milestone_rewards.id を参照）※外部キー制約なし |
| `stamp_count_used` | INTEGER | 使用したスタンプ数（後方互換のため残存） |
| `milestone_reached` | INTEGER | マイルストーン到達数（10, 20, 50など）**新規追加** |
| `is_milestone_based` | BOOLEAN | マイルストーン型かどうか（新仕様=true）**新規追加** |
| `valid_until` | TIMESTAMPTZ | 有効期限（自費メニュー用）**新規追加** |
| `is_first_time` | BOOLEAN | 初回特典かどうか（POIC本体込など）**新規追加** |
| `status` | TEXT | ステータス（pending: 未引渡, completed: 引渡済, cancelled: キャンセル, expired: 期限切れ） |
| `exchanged_at` | TIMESTAMPTZ | 交換申請日時 |
| `notes` | TEXT | 備考・スタッフ操作履歴を記録 |
| `created_at` | TIMESTAMPTZ | 作成日時 |

**重要な変更点**:
- ⚠️ `reward_id`の外部キー制約は削除されている（手動JOIN必須）
- ⚠️ `completed_at`, `completed_by` カラムは存在しない → `notes`にスタッフ操作履歴を記録
- ✅ `milestone_reached`で何個目のスタンプで獲得したかを記録

## 4. 機能仕様

### 4-1. 一覧表示

**API エンドポイント**: `GET /api/reward-exchanges`

**取得データ**:
```typescript
type RewardExchangeWithDetails = {
  id: string;
  user_id: string;
  user_name: string; // profiles.display_name
  user_picture_url: string | null; // profiles.picture_url
  reward_id: string;
  reward_name: string; // rewards.name
  reward_image_url: string | null; // rewards.image_url
  stamp_count_used: number;
  status: 'pending' | 'completed' | 'cancelled';
  exchanged_at: string; // ISO 8601
  notes: string | null;
  // NOTE: completed_at, completed_by カラムはDBに存在しない。status のみで管理。
};
```

**SQL クエリ（例）**:
```sql
SELECT
  re.id,
  re.user_id,
  p.display_name AS user_name,
  p.picture_url AS user_picture_url,
  re.reward_id,
  r.name AS reward_name,
  r.image_url AS reward_image_url,
  re.stamp_count_used,
  re.status,
  re.exchanged_at,
  re.notes
FROM reward_exchanges re
  INNER JOIN profiles p ON re.user_id = p.id
  INNER JOIN rewards r ON re.reward_id = r.id
ORDER BY re.exchanged_at DESC
LIMIT 50;
```

**フィルタ機能**:
- ステータスフィルタ: `?status=pending` / `?status=completed` / `?status=cancelled`
- 日付範囲フィルタ: `?from=2026-02-01&to=2026-02-28`
- 検索フィルタ: `?search=横山` （患者名または特典名で部分一致検索）

### 4-2. ステータス更新（引き渡し完了）

**API エンドポイント**: `POST /api/reward-exchanges/[id]/complete`

**リクエストボディ**: （2026-03-28更新：空でOK、スタッフ情報は自動取得）
```json
{}
```

**処理内容（2026-03-28更新）**:
1. セッションCookieからログイン中のスタッフ情報を自動取得
   - `staff.display_name` を取得（例: 「山田太郎」）
   - スタッフ情報が取得できない場合は「管理者」を使用
2. `reward_exchanges` の該当レコードを更新:
   - `status` を `completed` に変更
   - `notes` にスタッフ操作履歴を追記
     - フォーマット: `[2026-03-28T10:30:00.000Z] 山田太郎 が引き渡し完了`
   - 既存のnotesがある場合は改行して追記

**変更理由**: スタッフアカウント機能を活用し、担当者名の手動入力を不要に

### 4-3. ステータス更新（キャンセル）

**API エンドポイント**: `POST /api/reward-exchanges/[id]/cancel`

**リクエストボディ**:
```json
{
  "notes": "在庫切れのためキャンセル"
}
```

**処理内容**:
1. `reward_exchanges` の該当レコードを更新:
   - `status` を `cancelled` に変更

**⚠️ 重要**: スタンプは**返却しません**（積み上げ式のため）

### 4-4. 詳細モーダル（オプション）

交換履歴の行をクリックすると、詳細モーダルを表示:
- 患者情報（名前、プロフィール画像、診察券番号）
- 特典情報（特典名、画像、必要スタンプ数）
- 交換日時
- 引き渡し完了日時・担当者（completed の場合）
- 備考（cancelled の場合）

## 10. LIFF アプリとの連携仕様

### 10-1. LIFF側で実装すべき処理

患者がLIFFアプリで「この特典と交換する」ボタンを押した時：

```javascript
// 特典交換ボタンのクリック時
async function handleExchangeReward(rewardId, requiredStamps, rewardName) {
  const supabase = createClient();

  // reward_exchangesテーブルに記録を追加するだけ
  const { data, error } = await supabase
    .from('reward_exchanges')
    .insert({
      user_id: lineUserId,           // LINE User ID
      reward_id: rewardId,            // 特典ID（UUID）
      stamp_count_used: requiredStamps, // 必要スタンプ数（記録用のみ）
      status: 'pending',              // ステータスは必ず 'pending'
      exchanged_at: new Date().toISOString(),
      notes: `特典交換: ${rewardName}`, // 特典名を記録
    });

  // ⚠️ 重要: スタンプは減らしません（積み上げ式のため）
  // profiles.stamp_count は一切変更しない

  if (!error) {
    alert('特典交換を申請しました。受付でお受け取りください。');
  }
}
```

### 10-2. 重要な連携仕様

| 項目 | 仕様 |
|-----|-----|
| **ステータス初期値** | 必ず `'pending'`（未引渡）で登録 |
| **スタンプの扱い** | 交換しても**減らさない**（積み上げ式） |
| **stamp_count_used** | 記録用の数値（実際には減算しない） |
| **キャンセル時** | スタンプは元々減っていないので何もしない |
| **受け取り方法** | 受付で管理画面を見て手渡し → 「✓完了」ボタン |

### 10-3. データフロー

```
[患者] LIFF で交換申請
  ↓
[DB] reward_exchanges に status='pending' で記録
[DB] profiles.stamp_count は変更なし（積み上げ式）
  ↓
[受付] 管理画面で pending 一覧を確認
[受付] 患者来院時に特典を渡して「✓完了」クリック
  ↓
[DB] status='completed' に更新
```
