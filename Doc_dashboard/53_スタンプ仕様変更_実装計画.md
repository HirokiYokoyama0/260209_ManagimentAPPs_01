### Step 1: データベース準備

#### 1-1. stamp_method の拡張

**目的:** 購買インセンティブスタンプを区別できるようにする

**マイグレーションSQL:** `supabase/020_add_purchase_incentive_stamp_method.sql`

```sql
-- ========================================
-- スタンプ仕様変更: stamp_method に purchase_incentive を追加
-- ファイル: 020_add_purchase_incentive_stamp_method.sql
-- 作成日: 2026-03-26
-- 参照: Doc_dashboard/48_実装状況の確認結果_正しい仕様.md
-- ========================================

-- 既存の CHECK 制約を削除
ALTER TABLE stamp_history
DROP CONSTRAINT IF EXISTS stamp_history_stamp_method_check;

-- 新しい CHECK 制約を追加（purchase_incentive を含める）
-- 注: 現在の実装では "qr" と "qr_scan" の両方が使用されている
ALTER TABLE stamp_history
ADD CONSTRAINT stamp_history_stamp_method_check
CHECK (stamp_method IN (
  'qr',                 -- 現在の QR 読み取り実装
  'qr_scan',            -- 旧 QR 読み取り実装（後方互換性のため残存）
  'manual_admin',       -- 管理者による手動付与
  'import',             -- 一括インポート
  'survey_reward',      -- アンケート報酬
  'slot_game',          -- スロットゲーム
  'purchase_incentive'  -- 購買インセンティブQR読み取り（新規追加）
));

-- 確認用コメント
COMMENT ON COLUMN stamp_history.stamp_method IS
  'スタンプ付与方法: qr（来院QR）, qr_scan（旧来院QR・互換性）, manual_admin（管理者手動）, import（一括インポート）, survey_reward（アンケート報酬）, slot_game（スロットゲーム）, purchase_incentive（購買インセンティブQR）';
```

**実行方法:**
```bash
# Supabase CLI でマイグレーション実行
psql <接続文字列> -f supabase/020_add_purchase_incentive_stamp_method.sql
```

**確認SQL:**
```sql
-- CHECK 制約の確認
SELECT conname, pg_get_constraintdef(oid)
FROM pg_constraint
WHERE conrelid = 'stamp_history'::regclass
  AND conname = 'stamp_history_stamp_method_check';
```

#### 3-1. QRコードペイロード解析の更新

**対象ファイル:** （LIFFアプリ内の QR読み取り処理）

**必要な変更:**

```javascript
// 変更前: Premium=10, Regular=5 のみ対応
function parseQRPayload(qrData) {
  const payload = JSON.parse(qrData);

  if (payload.type === 'premium') {
    return { stamps: 10, method: 'qr_scan' };  // ❌ 古い値・旧 method
  } else if (payload.type === 'regular') {
    return { stamps: 5, method: 'qr_scan' };   // ❌ 古い値・旧 method
  }
}

// 変更後: Premium=15, Regular=10, Purchase=5 に対応
function parseQRPayload(qrData) {
  const payload = JSON.parse(qrData);

  if (payload.type === 'premium') {
    return { stamps: 15, method: 'qr', repeatable: false };      // ✅ 新しい値・現在の method
  } else if (payload.type === 'regular') {
    return { stamps: 10, method: 'qr', repeatable: false };      // ✅ 新しい値・現在の method
  } else if (payload.type === 'purchase') {
    return { stamps: 5, method: 'purchase_incentive', repeatable: true }; // ✅ 新規追加
  }
}
```

**注意:** 現在の実装では `stamp_method='qr'` を使用していますが、旧実装との互換性のため `'qr_scan'` も許可されています。新規実装では **`'qr'`** を使用してください。

#### 3-2. 繰り返しスキャン制御の実装

**要件:**
- Premium / Regular QR: 1日1回制限
- Purchase QR: 制限なし（何度でもスキャン可能）

**実装例:**

```javascript
async function handleQRScan(qrData) {
  const { stamps, method, repeatable } = parseQRPayload(qrData);
  const userId = getCurrentUserId();

  // 繰り返し不可のQRコード（来院QR）の場合
  if (!repeatable) {
    const today = new Date().toISOString().split('T')[0];
    const scannedToday = await checkStampHistory(userId, method, today);

    if (scannedToday) {
      alert('本日はすでにこのQRコードでスタンプを獲得済みです');
      return;
    }
  }

  // スタンプ付与API呼び出し
  await addStamps(userId, stamps, method);
  alert(`${stamps}スタンプを獲得しました！`);
}

// 今日既にスキャン済みかチェック
async function checkStampHistory(userId, stampMethod, date) {
  const response = await fetch(`/api/profiles/${userId}/stamp-history?date=${date}&method=${stampMethod}`);
  const data = await response.json();
  return data.length > 0;
}
```

#### 4-2. 二重スキャン防止（オプション・推奨）

**ファイル:** [`app/api/profiles/[id]/stamp/route.ts`](app/api/profiles/[id]/stamp/route.ts)

**追加実装:**

```typescript
// Line 18 の後に追加
// 来院QRの二重スキャン防止
// 注: 'qr' または 'qr_scan' の両方をチェック（後方互換性）
if ((stamp_method === 'qr' || stamp_method === 'qr_scan') && delta > 0) {
  const today = new Date().toISOString().split('T')[0];
  const { data: todayHistory } = await supabase
    .from('stamp_history')
    .select('id')
    .eq('user_id', id)
    .in('stamp_method', ['qr', 'qr_scan'])  // 両方をチェック
    .gte('visit_date', `${today}T00:00:00Z`)
    .lt('visit_date', `${today}T23:59:59Z`)
    .limit(1);

  if (todayHistory && todayHistory.length > 0) {
    return NextResponse.json(
      { error: '本日はすでに来院スタンプを獲得済みです' },
      { status: 400 }
    );
  }
}
// purchase_incentive の場合は制限なし（チェックをスキップ）
```

## ⚠️ リスクと対策

### リスク1: 既存QRコードの互換性

**問題:**
- 変更前に印刷・配布された QRコード（10/5スタンプ）が残っている

**対策案:**
- **Option A:** LIFF側で旧仕様もサポート（後方互換性）
  ```javascript
  // 旧仕様（stamps=10 or 5）と新仕様（stamps=15 or 10）の両方を受け入れる
  if (payload.type === 'premium') {
    const stamps = payload.stamps || 15; // デフォルト15
    // ...
  }
  ```
- **Option B:** 旧QRコードを全て回収・差し替え（運用コスト高）

### リスク2: 不正スキャン（Purchase QRの乱用）

**問題:**
- 購買インセンティブQRを悪意あるユーザーが繰り返しスキャンする可能性

**対策案:**
1. **運用ルール:** スタッフ立会い時のみスキャン可能にする
2. **技術的制限:** 1日あたりの上限回数を設定（例: 最大3回）
   ```javascript
   // 購買QRも1日3回まで制限
   if (method === 'purchase_incentive') {
     const todayCount = await getPurchaseScanCount(userId, today);
     if (todayCount >= 3) {
       alert('本日の購買スタンプは上限に達しました');
       return;
     }
   }
   ```
3. **動的QR生成:** QRコードに有効期限やトークンを持たせる（将来対応）

### リスク3: LIFF アプリとの連携遅延

**問題:**
- LIFF アプリが別リポジトリで開発されており、リリースタイミングがずれる可能性

**対策:**
- ダッシュボード側の変更を先行リリースしても問題ないか確認
- LIFF側のリリースまで新QRコードを運用に投入しない
