# スタンプ仕様変更 影響範囲分析

## 📋 変更概要

### 現行仕様
- **優良患者様（Premium）:** 10スタンプ（来院時1回のみ）
- **通常患者様（Regular）:** 5スタンプ（来院時1回のみ）

### 変更後の仕様
- **優良患者様（Premium）:** 15スタンプ（来院時1回のみ）
- **通常患者様（Regular）:** 10スタンプ（来院時1回のみ）
- **購買インセンティブ（Purchase Incentive）:** 5スタンプ（来院時何度でも）← **新規追加**

## 🔄 繰り返しスキャン制御の実装方針

### 要件整理

| QRコード種別 | 1日あたりのスキャン制限 |
|--------------|------------------------|
| 優良患者様（Premium） | 1回のみ |
| 通常患者様（Regular） | 1回のみ |
| 購買インセンティブ（Purchase） | **制限なし（何度でも）** |

### 実装方法

#### LIFF アプリ側での制御
```javascript
// 擬似コード
async function handleQRScan(payload) {
  const { type, stamps } = JSON.parse(payload);

  if (type === 'purchase') {
    // 購買インセンティブ: 制限なし
    await addStamps(stamps, 'purchase_incentive');
  } else {
    // Premium / Regular: 今日スキャン済みかチェック
    const scannedToday = await checkTodayStampHistory(type);
    if (scannedToday) {
      alert('本日はすでにスタンプを獲得済みです');
      return;
    }
    await addStamps(stamps, 'qr');  // 現在の実装は 'qr'
  }
}
```

#### API側での二重スキャン防止（参考）
```typescript
// app/api/stamps/auto/route.ts または app/api/stamps/scan/route.ts での検証例
// 注: 現在の実装では stamp_method = 'qr' を使用
if (stampMethod === 'qr') {
  // 来院QRは1日1回制限
  const today = new Date().toISOString().split('T')[0];
  const { data: todayHistory } = await supabase
    .from('stamp_history')
    .select('id')
    .eq('user_id', userId)
    .eq('stamp_method', 'qr')
    .gte('visit_date', today + 'T00:00:00Z')
    .limit(1);

  if (todayHistory && todayHistory.length > 0) {
    return NextResponse.json(
      { error: '本日はすでに来院スタンプを獲得済みです' },
      { status: 400 }
    );
  }
}
// purchase_incentive の場合は制限なし
```

**注:** 旧実装（`stamp_method='qr_scan'`）との互換性を考慮する場合は、以下のように両方をチェックする：

```typescript
// 旧実装との互換性を考慮した実装例
const { data: todayHistory } = await supabase
  .from('stamp_history')
  .select('id')
  .eq('user_id', userId)
  .in('stamp_method', ['qr', 'qr_scan'])  // 両方をチェック
  .gte('visit_date', today + 'T00:00:00Z')
  .limit(1);
```

## ⚠️ 注意事項・懸念点

### 1. 積み上げ式の原則は維持
- スタンプは加算のみ（減算しない）
- 購買インセンティブも同様に加算のみ
- 参照: [`Doc_dashboard/10_重要_スタンプ仕様_積み上げ式.md`](Doc_dashboard/10_重要_スタンプ仕様_積み上げ式.md)

### 2. 不正スキャン対策
- **課題:** 購買インセンティブQRコードを悪意あるユーザーが繰り返しスキャンする可能性
- **対策案:**
  - ① スタッフ立会い時のみスキャン可能にする（運用ルール）
  - ② 1日あたりの上限回数を設定（例: 1日最大3回まで）
  - ③ QRコードに有効期限を持たせる（動的QR生成）
  - ④ スキャン時にスタッフ承認を必須にする（LIFF側で実装）

### 3. 既存QRコードの取り扱い
- **問題:** 変更前に印刷・配布された古いQRコード（10/5スタンプ）が残っている可能性
- **対応:**
  - LIFF側で旧仕様も一定期間サポート（後方互換性）
  - または、古いQRコードを全て回収・差し替え（運用コスト高）

### 4. LIFF アプリ側の実装確認
- **要確認項目:**
  - `type: "purchase"` の処理が実装されているか
  - 繰り返しスキャン制御のロジックが正しく動作するか
  - API リクエスト時に `stamp_method` を正しく指定しているか
