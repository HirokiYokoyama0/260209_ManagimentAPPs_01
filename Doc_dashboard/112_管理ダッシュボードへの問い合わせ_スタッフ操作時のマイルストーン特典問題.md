# 管理ダッシュボードへの問い合わせ: スタッフ操作時のマイルストーン特典問題

**作成日:** 2026-04-04
**作成者:** LIFFアプリ開発チーム (Claude Code)
**宛先:** 管理ダッシュボード開発チーム
**件名:** スタッフ操作時のマイルストーン特典の挙動について

---

## 📋 背景

LIFFアプリ側でマイルストーン特典システムを実装中、スタッフ操作時の特典の挙動について以下の問題が報告されました:

### 報告された問題

1. **スタンプ減少時の特典残存**
   - スタッフ操作でスタンプ数を0に減らす
   - マイルストーン特典が「受付確認中」(pending) のまま残る
   - 期待: status が 'cancelled' になる

2. **スタンプ増加時の自動付与**
   - 管理ダッシュボードでスタンプ数を増やす
   - マイルストーン特典が自動的に「受付確認中」で付与される
   - 期待: スタッフ操作では自動付与されない

3. **マイルストーン特典の削除失敗**
   - スタッフ操作でスタンプを0にする
   - マイルストーン特典が削除されない
   - 期待: 全てのマイルストーン特典が削除（cancelled）される

---

## 🔍 LIFFアプリ側の調査結果

### 現在の実装（ミニアプリ側）

**エンドポイント:** `POST /api/stamps/manual`
**使用箇所:** LIFFアプリのスタッフ操作画面

**処理フロー:**
```typescript
1. スタンプ数を取得 (currentStampCount)
2. 新しいスタンプ数を設定 (newStampCount)
3. changeAmount = newStampCount - currentStampCount を計算
4. stamp_history の全レコードを削除（起点としてリセット）
5. stamp_history に新しいレコードを INSERT
6. profiles.stamp_count を直接 UPDATE
7. changeAmount < 0 の場合のみ:
   - invalidateMilestoneRewards() を呼ぶ（特典を 'cancelled' に変更）
8. changeAmount > 0 の場合:
   - 何もしない（マイルストーン判定も特典付与もしない）
```

**重要な点:**
- スタンプ**減少時**: マイルストーン特典を `cancelled` に変更
- スタンプ**増加時**: マイルストーン特典を**自動付与しない**

## ❓ 管理ダッシュボード側への質問

### 質問1: 使用しているAPIエンドポイント

**質問:**
- 管理ダッシュボードでスタンプ数を変更する際、どのAPIエンドポイントを使用していますか？

**選択肢:**
- A. LIFFアプリ側の `POST /api/stamps/manual` を直接使用
- B. 管理ダッシュボード専用の別のAPIエンドポイントを使用
- C. Supabaseに直接アクセスして `profiles.stamp_count` を UPDATE

**背景:**
- LIFFアプリ側の `/api/stamps/manual` ではスタンプ増加時にマイルストーン特典を自動付与しない実装になっている
- しかし「管理ダッシュボードでスタンプ数を増やすと特典が自動付与される」という報告がある
- 別のAPIエンドポイントを使用している可能性がある

### 質問2: マイルストーン判定の実装

**質問:**
- 管理ダッシュボード側でスタンプ数を変更した際、マイルストーン判定（checkMilestones）を実行していますか？

**確認事項:**
- `checkMilestones()` 関数を呼んでいるか？
- `grantMilestoneReward()` 関数を呼んでいるか？
- マイルストーン特典を自動的に `reward_exchanges` テーブルに INSERT しているか？

**背景:**
- 問題2の原因を特定するため
- LIFFアプリ側と管理ダッシュボード側で実装が異なる可能性がある

### 質問3: スタンプ減少時の挙動

**質問:**
- 管理ダッシュボードでスタンプ数を減らした際、マイルストーン特典の無効化処理を実行していますか？

**確認事項:**
- `invalidateMilestoneRewards()` 関数を呼んでいるか？
- または、`reward_exchanges` テーブルの status を直接 UPDATE しているか？

**背景:**
- 問題1・3の原因を特定するため
- LIFFアプリ側では `invalidateMilestoneRewards()` を実装済み

## 💡 推奨される実装方針

### 方針A: LIFFアプリ側のAPIを使用（推奨）

**利点:**
- 実装が統一される
- マイルストーン特典の無効化処理が自動的に実行される
- バグが減る

**実装:**
```typescript
// 管理ダッシュボード側
const response = await fetch('/api/stamps/manual', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    userId: 'U5c70cd61f4fe89a65381cd7becee8de3',
    staffPin: '****',
    newStampCount: 0
  })
});
```

**動作:**
- スタンプ減少時: 自動的にマイルストーン特典を `cancelled` に変更
- スタンプ増加時: マイルストーン特典を自動付与**しない**

### 方針B: 管理ダッシュボード専用APIを実装（非推奨）

**利点:**
- 管理ダッシュボード側で独自の制御が可能

**欠点:**
- 実装が分散する
- バグが発生しやすい
- メンテナンスコストが増加

**実装が必要な処理:**
1. スタンプ減少時: `invalidateMilestoneRewards()` を呼ぶ
2. スタンプ増加時: マイルストーン判定を**呼ばない**（手動付与にする）

## 🎯 仕様の確認

### スタッフ操作時のマイルストーン特典の扱い

**質問:**
- スタッフ操作でスタンプ数を増やした場合、マイルストーン特典を自動付与すべきですか？

**選択肢:**
- **A. 自動付与しない（推奨）**
  - 理由: スタッフ操作は特殊な状況（エラー修正、クレーム対応など）が多い
  - 理由: 意図しない特典付与を防ぐ
  - 代替: 必要であれば、管理ダッシュボードで手動で特典を付与する機能を追加

- **B. 自動付与する**
  - 理由: ユーザーに不公平がないようにする
  - 注意: QRスキャンと同じ挙動になるため、スタッフ操作の意味が薄れる

**現在のLIFFアプリ側の実装:** A（自動付与しない）
