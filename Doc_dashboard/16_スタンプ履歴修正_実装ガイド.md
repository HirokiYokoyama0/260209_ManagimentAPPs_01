## 修正の概要

### 目的

管理ダッシュボードでスタンプ数を変更した際に、`stamp_history` テーブルに履歴を正しく記録し、データ整合性を保つ。

### 修正前の問題

**❌ 修正前:**
```typescript
// profiles を直接更新するだけ
await supabase.from("profiles").update({ stamp_count: newCount });

// ❌ stamp_history には何も記録されない
// ❌ トリガーが発火しない
// ❌ データ不整合が発生
```

**結果:**
- `profiles.stamp_count` と `stamp_history` の MAX(stamp_number) が不一致
- 完全な履歴が残らない
- トリガーの設計思想と矛盾

### 修正後の動作

**✅ 修正後:**
```typescript
// 1. スタンプ減少時は過去レコードを削除
if (newCount < currentStampCount) {
  await supabaseAdmin.from("stamp_history").delete()
    .eq("user_id", id)
    .gt("stamp_number", newCount);
}

// 2. stamp_history に履歴を記録
await supabase.from("stamp_history").insert({
  user_id: id,
  stamp_number: newCount,
  amount: changeAmount,
  stamp_method: "manual_admin",
  // ...
});

// 3. トリガーが自動的に profiles を更新
// 4. 念のため手動でも profiles を更新（安全策）
```

**結果:**
- ✅ データ整合性が保たれる
- ✅ 完全な履歴が残る
- ✅ トリガーが正しく動作

## 問題の詳細

### 背景

`stamp_history` テーブルのトリガーは `MAX(stamp_number)` で `profiles.stamp_count` を計算する設計です。

```sql
-- トリガー関数
UPDATE profiles
SET stamp_count = (
  SELECT COALESCE(MAX(stamp_number), 0)
  FROM stamp_history
  WHERE user_id = NEW.user_id
)
WHERE id = NEW.user_id;
```

### 問題のシナリオ

**ケース1: スタンプを増やす場合（問題なし）**
```
1. 現在: profiles.stamp_count = 50
2. 管理者が 60 に変更
3. profiles を直接更新 → stamp_count = 60
4. stamp_history には記録されない
5. 結果: 見た目は60だが、MAX(stamp_history.stamp_number) = 50
```

**ケース2: スタンプを減らす場合（深刻な問題）**
```
1. 現在: profiles.stamp_count = 63
2. stamp_history に stamp_number = 63 のレコードが存在
3. 管理者が 56 に変更
4. 新しい stamp_history レコードを INSERT（stamp_number = 56）
5. トリガー実行: stamp_count = MAX(63, 56) = 63
6. 結果: 画面上は56に見えるが、タブ切替で63に戻る ❌
```

### 根本原因

トリガーの計算方法が異なる：
- **INSERT トリガー**: `MAX(stamp_number)` で計算
- **DELETE トリガー**: `SUM(amount)` で計算

この不整合により、スタンプ減少時に過去のレコードが残っていると、MAX() が古い値を返してしまう。

## 実装内容

### 修正したファイル

1. **`app/api/profiles/[id]/stamp-set/route.ts`**
   - スタンプ数を直接設定するAPI
   - 管理画面の「スタンプ数変更」機能で使用

2. **`app/api/profiles/[id]/stamp/route.ts`**
   - スタンプ数を±1増減するAPI
   - 現在は使用されていない可能性があるが、念のため修正

### 実装のポイント

#### 1️⃣ SERVICE_ROLE_KEY の使用

```typescript
import { createSupabaseAdminClient } from "@/lib/supabase/server-admin";

const supabaseAdmin = createSupabaseAdminClient();
```

**理由:**
- `stamp_history` テーブルの RLS ポリシーは DELETE を許可していない
- ANON_KEY では削除できないため、SERVICE_ROLE_KEY が必須

#### 2️⃣ スタンプ減少時の過去レコード削除

```typescript
if (newCount < currentStampCount) {
  // stamp_number が新しい値より大きいレコードを削除
  await supabaseAdmin
    .from("stamp_history")
    .delete({ count: "exact" })
    .eq("user_id", id)
    .gt("stamp_number", newCount);
}
```

**理由:**
- MAX() 計算で古い値が返らないようにする
- データ整合性を保つ

#### 3️⃣ stamp_history に履歴を記録

```typescript
await supabase.from("stamp_history").insert({
  user_id: id,
  visit_date: new Date().toISOString(),
  stamp_number: newCount,        // 変更後の値
  amount: changeAmount,           // 差分
  stamp_method: "manual_admin",
  qr_code_id: `admin-${Date.now()}`,
  notes: "管理ダッシュボードからスタンプ数を手動変更",
});
```

#### 4️⃣ 手動での profiles 更新（安全策）

```typescript
await supabaseAdmin
  .from("profiles")
  .update({
    stamp_count: newCount,
    updated_at: new Date().toISOString(),
  })
  .eq("id", id);
```

**理由:**
- トリガーの計算方法の不整合対策
- トリガーは動作するが、念のため手動でも更新

#### 5️⃣ activity_logs に記録

```typescript
await logActivityIfStaff(request, "stamp_set", {
  targetType: "profile",
  targetId: id,
  details: {
    old_value: currentStampCount,
    new_value: newCount,
    delta: changeAmount,
  },
});
```

## データ整合性確認

### すべての患者の整合性をチェック

Supabase SQL エディタで以下のクエリを実行：

```sql
-- 不整合を確認
SELECT
  p.id AS user_id,
  p.display_name,
  p.stamp_count AS profile_stamp_count,
  COALESCE(MAX(sh.stamp_number), 0) AS history_max_stamp_number,
  p.stamp_count - COALESCE(MAX(sh.stamp_number), 0) AS difference
FROM profiles p
LEFT JOIN stamp_history sh ON p.id = sh.user_id
GROUP BY p.id, p.display_name, p.stamp_count
HAVING p.stamp_count != COALESCE(MAX(sh.stamp_number), 0);
```

**期待される結果:**
- 0件（不整合なし）✅

**不整合がある場合:**
- 修正前のデータが残っている可能性
- 手動で修正が必要

---

### 修正前のデータを修正する（必要な場合）

不整合があるユーザーに対して、手動で `stamp_history` を追加：

```sql
-- 例: user_id = 'Uxxxx', profiles.stamp_count = 63, MAX(stamp_history.stamp_number) = 50
INSERT INTO stamp_history (user_id, visit_date, stamp_number, amount, stamp_method, notes)
VALUES (
  'Uxxxx',
  NOW(),
  63,  -- profiles.stamp_count の値
  13,  -- 差分（63 - 50）
  'manual_fix',
  '不整合修正：管理画面での手動変更を遡及記録'
);
```
