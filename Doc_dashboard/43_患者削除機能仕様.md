## 概要

### 目的

管理ダッシュボードから患者情報を削除する機能を提供する。

### 主な要件

- ✅ 患者（profiles）の完全削除
- ✅ 関連データの整合性を保つ（孤立レコード防止）
- ✅ 家族構成への影響を最小化
- ✅ 誤削除防止（確認ダイアログ、権限チェック）
- ✅ 削除ログの記録（監査用）

### 想定ユースケース

1. **テストユーザーの削除**: 開発・検証用に作成したユーザーの削除
2. **誤登録の削除**: 重複アカウント、登録ミスの削除
3. **退会希望**: 患者本人が退会を希望した場合
4. **長期未来院**: 数年以上来院がなく、連絡も取れない患者

## 影響範囲の分析

### 📊 データベース全体への影響マップ

```
profiles (削除対象)
│
├─ CASCADE DELETE (自動削除される)
│  ├─ stamp_history ※1
│  ├─ reward_exchanges ※2
│  ├─ patient_dental_records ※3
│  └─ event_logs (user_id = 削除対象) ※4
│
├─ SET NULL (NULL になる)
│  ├─ families.representative_user_id ※5
│  └─ activity_logs.target_id (間接参照) ※6
│
└─ MANUAL HANDLING (手動対応が必要)
   ├─ profiles.family_id (家族メンバーの再配置) ※7
   └─ family_stamp_totals (VIEW - 自動再計算) ※8
```

### ※1 stamp_history テーブル（CASCADE DELETE）

**外部キー制約:**
```sql
FOREIGN KEY: user_id → profiles(id) ON DELETE CASCADE
```

**影響:**
- 患者を削除すると、その患者のスタンプ履歴は**すべて自動削除**される
- 復元不可能

**対応:**
- 削除前に履歴データをエクスポート機能を提供（オプション）
- 確認ダイアログで「スタンプ履歴 X 件」を表示

---

### ※2 reward_exchanges テーブル（CASCADE DELETE）

**外部キー制約:**
```sql
FOREIGN KEY: user_id → profiles(id) ON DELETE CASCADE
```

**影響:**
- 患者を削除すると、その患者の特典交換履歴は**すべて自動削除**される
- 統計データに影響（交換回数が減る）

**対応:**
- 削除前に確認ダイアログで「特典交換履歴 X 件」を表示
- 削除実行前に `activity_logs` に「削除予定の履歴情報」をJSON形式で保存

---

### ※3 patient_dental_records テーブル（CASCADE DELETE）

**外部キー制約:**
```sql
FOREIGN KEY: patient_id → profiles(id) ON DELETE CASCADE
```

**影響:**
- 患者を削除すると、その患者のケア記録は**すべて自動削除**される
- 治療履歴が失われる（医療記録の観点から重要）

**対応:**
- 削除前に確認ダイアログで「ケア記録 X 件」を表示
- **重要:** 医療記録の保存義務を考慮し、削除前にエクスポート機能を提供（推奨）
- 削除実行前に `activity_logs` に「削除予定のケア記録情報」をJSON形式で保存

---

### ※4 event_logs テーブル（SET NULL または KEEP）

**外部キー制約:**
```sql
FOREIGN KEY: user_id → profiles(id) (RLS ポリシーのみで制約なし)
```

**影響:**
- `event_logs` には外部キー制約が**設定されていない**
- 患者を削除しても、`event_logs` の `user_id` は削除されたIDのまま残る（孤立データ）
- 分析データとしては価値があるため、ログは残す方針

**対応:**
- **ログは削除しない**（分析・監査用）
- 将来的に `profiles` テーブルに論理削除フラグ（`is_deleted`）を追加し、物理削除は避ける方針も検討

---

### ※5 families.representative_user_id（SET NULL）

**外部キー制約:**
```sql
FOREIGN KEY: representative_user_id → profiles(id) ON DELETE SET NULL
```

**影響:**
- 削除対象が**家族の代表者（parent）**の場合、`families.representative_user_id` が **NULL** になる
- 家族は残る（家族名、他のメンバーはそのまま）
- 代表者がいない家族が発生する

**対応:**

#### 📌 ケース1: 削除対象が単身家族（member_count = 1）
→ **家族ごと削除**
```sql
DELETE FROM families WHERE id = 'family_id';
```

#### 📌 ケース2: 削除対象が代表者で、他にメンバーがいる（member_count >= 2）
→ **新しい代表者を自動選出**
```sql
-- 家族内の他のメンバー（最も古い created_at のユーザー）を新代表者にする
UPDATE families
SET representative_user_id = (
  SELECT id FROM profiles
  WHERE family_id = 'family_id' AND id != '削除対象ID'
  ORDER BY created_at ASC
  LIMIT 1
)
WHERE id = 'family_id';
```

#### 📌 ケース3: 削除対象が子メンバー（child）
→ **何もしない**（代表者はそのまま）

---

### ※6 activity_logs.target_id（間接参照）

**外部キー制約:**
```sql
-- 外部キー制約なし（TEXT型で自由に格納）
```

**影響:**
- `activity_logs` の `target_id` には削除されたプロフィールのIDが残る（孤立データ）
- ログなので削除しない方針

**対応:**
- **ログは削除しない**（監査用）
- 削除時に `action = 'profile_deleted'` のログを追加

---

### ※7 profiles.family_id（家族メンバーの再配置）

**外部キー制約:**
```sql
FOREIGN KEY: family_id → families(id) ON DELETE SET NULL
```

**影響:**
- 家族を削除すると、その家族に所属していたメンバーの `family_id` が **NULL** になる（単身に戻る）

**対応:**

#### パターンA: 単身家族を削除する場合
→ 通常削除（profiles 削除 → families 削除）

#### パターンB: 複数メンバーの家族を削除する場合
→ **削除前に全メンバーを単身家族に分離**
```sql
-- 各メンバーに新しい単身家族を作成
FOR EACH member IN family_members LOOP
  INSERT INTO families (family_name, representative_user_id)
  VALUES (member.display_name || 'の家族', member.id)
  RETURNING id INTO new_family_id;

  UPDATE profiles
  SET family_id = new_family_id, family_role = 'parent'
  WHERE id = member.id;
END LOOP;

-- 元の家族を削除
DELETE FROM families WHERE id = old_family_id;
```

---

### ※8 family_stamp_totals（VIEW - 自動再計算）

**タイプ:** ビュー（リアルタイム計算）

**影響:**
- プロフィールや家族が削除されると、**ビューは自動的に再計算**される
- データ整合性は自動的に保たれる

**対応:**
- 何もしなくて OK（ビューは自動更新）

## 削除方式の選択

### 方式1: 物理削除（Hard Delete） ← **採用**

**定義:** データベースから完全に削除（`DELETE FROM profiles WHERE id = ?`）

**メリット:**
- ✅ データベースが肥大化しない
- ✅ 個人情報保護の観点で優れている（GDPR等に対応）
- ✅ 実装がシンプル

**デメリット:**
- ❌ 復元不可能
- ❌ 統計データへの影響（スタンプ合計数が減る）
- ❌ ログの `user_id` が孤立する

**採用理由:**
- 歯科医院の患者管理システムでは、誤削除を防ぐ確認ダイアログを実装すれば十分
- 個人情報保護の観点から、不要なデータは完全削除が望ましい
- 削除前に `activity_logs` に詳細情報を JSON 形式で保存することで、監査証跡を残す

---

### 方式2: 論理削除（Soft Delete）

**定義:** `profiles.is_deleted = TRUE` のようなフラグを追加し、削除済みとして扱う

**メリット:**
- ✅ 復元が可能
- ✅ 統計データへの影響なし
- ✅ 監査証跡が完璧

**デメリット:**
- ❌ データベースが肥大化
- ❌ 個人情報が残り続ける（GDPR 等に非対応）
- ❌ すべてのクエリに `WHERE is_deleted = FALSE` が必要（複雑化）

**不採用理由:**
- 患者情報という個人情報を削除せずに残すことはリスクが高い
- 論理削除は主にビジネスデータ（注文履歴など）で使われる手法で、個人情報管理には不向き

## 実装仕様

### 削除フロー

```
1. 管理ダッシュボードで患者を選択
   ↓
2. 「削除」ボタンをクリック
   ↓
3. 確認ダイアログを表示
   ├─ 患者名・診察券番号
   ├─ 削除される関連データの件数
   │  ├─ スタンプ履歴: X 件
   │  ├─ 特典交換履歴: X 件
   │  ├─ ケア記録: X 件
   │  └─ 家族情報: "○○家の代表者" or "○○家のメンバー"
   └─ ⚠️ 警告: 「この操作は取り消せません」
   ↓
4. パスワード再入力 or 「削除する」と入力（誤操作防止）
   ↓
5. DELETE /api/admin/profiles/[id] を呼び出し
   ↓
6. バックエンドで削除処理実行
   ├─ ステップ1: 関連データ件数を取得（ログ記録用）
   ├─ ステップ2: activity_logs に削除予定データを記録
   ├─ ステップ3: 家族構成の調整
   │  ├─ 単身家族 → 家族ごと削除
   │  ├─ 代表者 → 新代表者を選出 or 家族解散
   │  └─ 子メンバー → 何もしない
   ├─ ステップ4: profiles を DELETE（CASCADE で関連データ自動削除）
   └─ ステップ5: activity_logs に削除完了ログを記録
   ↓
7. 削除完了メッセージを表示
   ↓
8. 患者リストをリフレッシュ
```

### バックエンド実装（Next.js API）

**ファイル:** `app/api/admin/profiles/[id]/route.ts`

```typescript
import { NextRequest, NextResponse } from "next/server";
import { createSupabaseServerClient } from "@/lib/supabase/server";

export async function DELETE(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const profileId = params.id;

  try {
    const supabase = await createSupabaseServerClient();

    // ステップ1: 削除対象のプロフィールを取得
    const { data: profile, error: profileError } = await supabase
      .from("profiles")
      .select("*, families!inner(id, family_name, representative_user_id)")
      .eq("id", profileId)
      .single();

    if (profileError || !profile) {
      return NextResponse.json(
        { error: "Profile not found" },
        { status: 404 }
      );
    }

    // ステップ2: 関連データ件数を取得（ログ記録用）
    const [stampHistoryCount, rewardExchangesCount, dentalRecordsCount] =
      await Promise.all([
        supabase
          .from("stamp_history")
          .select("id", { count: "exact", head: true })
          .eq("user_id", profileId),
        supabase
          .from("reward_exchanges")
          .select("id", { count: "exact", head: true })
          .eq("user_id", profileId),
        supabase
          .from("patient_dental_records")
          .select("id", { count: "exact", head: true })
          .eq("patient_id", profileId),
      ]);

    const deletedCounts = {
      stampHistory: stampHistoryCount.count || 0,
      rewardExchanges: rewardExchangesCount.count || 0,
      dentalRecords: dentalRecordsCount.count || 0,
    };

    // ステップ3: activity_logs に削除予定データを記録
    await supabase.from("activity_logs").insert({
      action: "profile_delete_initiated",
      target_type: "profile",
      target_id: profileId,
      details: {
        profile: {
          id: profile.id,
          display_name: profile.display_name,
          real_name: profile.real_name,
          ticket_number: profile.ticket_number,
          stamp_count: profile.stamp_count,
          visit_count: profile.visit_count,
        },
        family: profile.families,
        relatedDataCounts: deletedCounts,
      },
    });

    // ステップ4: 家族構成の調整
    let familyAction: "family_deleted" | "new_representative_assigned" | "no_change" = "no_change";
    const familyId = profile.family_id;
    const isRepresentative = profile.families?.representative_user_id === profileId;

    if (familyId) {
      // 家族のメンバー数を取得
      const { data: familyMembers } = await supabase
        .from("profiles")
        .select("id, display_name, created_at")
        .eq("family_id", familyId);

      const memberCount = familyMembers?.length || 0;

      if (memberCount === 1) {
        // 単身家族 → 家族ごと削除
        await supabase.from("families").delete().eq("id", familyId);
        familyAction = "family_deleted";
      } else if (isRepresentative && memberCount >= 2) {
        // 代表者削除 → 新代表者を選出
        const newRepresentative = familyMembers
          ?.filter((m) => m.id !== profileId)
          .sort((a, b) => new Date(a.created_at).getTime() - new Date(b.created_at).getTime())[0];

        if (newRepresentative) {
          await supabase
            .from("families")
            .update({ representative_user_id: newRepresentative.id })
            .eq("id", familyId);

          await supabase
            .from("profiles")
            .update({ family_role: "parent" })
            .eq("id", newRepresentative.id);

          familyAction = "new_representative_assigned";
        }
      }
    }

    // ステップ5: profiles を削除（CASCADE で関連データ自動削除）
    const { error: deleteError } = await supabase
      .from("profiles")
      .delete()
      .eq("id", profileId);

    if (deleteError) {
      throw new Error(`Failed to delete profile: ${deleteError.message}`);
    }

    // ステップ6: activity_logs に削除完了ログを記録
    await supabase.from("activity_logs").insert({
      action: "profile_deleted",
      target_type: "profile",
      target_id: profileId,
      details: {
        deletedProfile: {
          id: profile.id,
          display_name: profile.display_name,
          ticket_number: profile.ticket_number,
        },
        deletedRelatedData: {
          ...deletedCounts,
          family: {
            deleted: familyAction === "family_deleted",
            familyName: profile.families?.family_name || null,
            action: familyAction,
          },
        },
      },
    });

    // レスポンス返却
    return NextResponse.json({
      success: true,
      deletedProfile: {
        id: profile.id,
        display_name: profile.display_name,
        ticket_number: profile.ticket_number,
      },
      deletedRelatedData: {
        ...deletedCounts,
        family: {
          deleted: familyAction === "family_deleted",
          familyName: profile.families?.family_name || null,
          action: familyAction,
        },
      },
    });
  } catch (error: any) {
    console.error("Error deleting profile:", error);
    return NextResponse.json(
      { error: "Failed to delete profile", details: error.message },
      { status: 500 }
    );
  }
}
```

### 既存の外部キー制約（確認）

現在のデータベース設計では、以下の制約が設定されています：

## フロントエンド実装

### 確認ダイアログコンポーネント

**ファイル:** `components/admin/delete-profile-dialog.tsx`

```typescript
"use client";

import { useState } from "react";
import { Button } from "@/components/ui/button";
import {
  Dialog,
  DialogContent,
  DialogDescription,
  DialogFooter,
  DialogHeader,
  DialogTitle,
  DialogTrigger,

import { Trash2, AlertTriangle } from "lucide-react";
import { toast } from "sonner";

interface DeleteProfileDialogProps {
  profile: {
    id: string;
    display_name: string;
    real_name?: string;
    ticket_number?: string;
    stamp_count: number;
    visit_count: number;

                <span className="font-medium">来院回数:</span>{" "}
                {profile.visit_count} 回
              </p>
            </div>
          </div>

          {/* 削除される関連データ */}
          <div className="bg-red-50 p-4 rounded-lg border border-red-200">
            <h3 className="font-semibold text-red-900 mb-2">
              削除される関連データ
            </h3>
            <ul className="space-y-1 text-sm text-red-800">
              <li>• スタンプ履歴: {relatedCounts.stampHistory} 件</li>
              <li>• 特典交換履歴: {relatedCounts.rewardExchanges} 件</li>
              <li>• ケア記録: {relatedCounts.dentalRecords} 件</li>
              {relatedCounts.familyInfo && (
                <li>
                  • 家族情報: {relatedCounts.familyInfo.family_name}{" "}
                  {relatedCounts.familyInfo.is_representative
                    ? "(代表者)"
                    : "(メンバー)"}
                  {relatedCounts.familyInfo.member_count === 1 &&
                    " → 家族ごと削除"}
                  {relatedCounts.familyInfo.is_representative &&
                    relatedCounts.familyInfo.member_count > 1 &&
                    " → 新代表者を自動選出"}
                </li>
              )}
            </ul>
          </div>

          {/* 確認入力 */}
          <div>
            <Label htmlFor="confirmation">
              削除を実行するには「<strong>削除する</strong>」と入力してください
            </Label>
            <Input
              id="confirmation"
              value={confirmation}
              onChange={(e) => setConfirmation(e.target.value)}
              placeholder="削除する"
              className="mt-2"
            />
          </div>

          {/* 警告メッセージ */}
          <Alert variant="destructive">
            <AlertTriangle className="h-4 w-4" />
            <AlertDescription>
              この操作は取り消せません。データは完全に削除され、復元できません。
            </AlertDescription>
          </Alert>
        </div>

        <DialogFooter>
          <Button variant="outline" onClick={() => setOpen(false)}>
            キャンセル
          </Button>
          <Button
            variant="destructive"
            onClick={handleDelete}
            disabled={!isConfirmed || loading}
          >
            {loading ? "削除中..." : "削除する"}
          </Button>
        </DialogFooter>
      </DialogContent>
    </Dialog>
  );
}
```

## LIFF開発者への伝達事項

### 📢 重要: 患者削除機能の実装について

#### 1. オンボーディングフローの実装

**背景:**
- 管理ダッシュボードから患者情報が削除されると、Supabase の `profiles` テーブルからデータが完全に削除されます
- 削除された患者が再度 LIFF アプリにアクセスした場合、`profiles` テーブルにデータが存在しないため、**新規ユーザー**として扱う必要があります

**必須対応:**

```typescript
// LIFF アプリの初期化時
const liff = (await import("@line/liff")).default;
await liff.init({ liffId: process.env.NEXT_PUBLIC_LIFF_ID! });

if (liff.isLoggedIn()) {
  const profile = liff.getProfile();
  const lineUserId = profile.userId;

  // Supabase で profiles テーブルを確認
  const supabase = createClient();
  const { data: existingProfile, error } = await supabase
    .from("profiles")
    .select("*")
    .eq("id", lineUserId)
    .single();

  if (error || !existingProfile) {
    // ✅ プロフィールが存在しない場合 → オンボーディングへ
    console.log("Profile not found. Redirecting to onboarding...");
    router.push("/onboarding");
  } else {
    // ✅ プロフィールが存在する場合 → ホーム画面へ
    router.push("/home");
  }
}
```

#### 2. オンボーディング画面の実装内容

**ページ:** `/onboarding`

**必要な処理:**
1. LINE プロフィール情報を取得（表示名、プロフィール画像）
2. 新規 `profiles` レコードを作成
3. 新規 `families` レコードを作成（単身家族として）
4. `profiles.family_id` を設定

**実装例:**

```typescript
// pages/onboarding.tsx
const handleOnboarding = async () => {
  const liff = (await import("@line/liff")).default;
  const profile = await liff.getProfile();

  const supabase = createClient();

  // 1. 新規家族を作成
  const { data: newFamily, error: familyError } = await supabase
    .from("families")
    .insert({
      family_name: `${profile.displayName}の家族`,
      representative_user_id: profile.userId,
    })
    .select()
    .single();

  if (familyError) {
    console.error("Failed to create family:", familyError);
    return;
  }

  // 2. 新規プロフィールを作成
  const { error: profileError } = await supabase.from("profiles").insert({
    id: profile.userId,
    line_user_id: profile.userId,
    display_name: profile.displayName,
    picture_url: profile.pictureUrl,
    family_id: newFamily.id,
    family_role: "parent",
    stamp_count: 0,
    visit_count: 0,
  });

  if (profileError) {
    console.error("Failed to create profile:", profileError);
    return;
  }

  // 3. ホーム画面へリダイレクト
  router.push("/home");
};
```

#### 4. データ削除後の再登録について

**重要な注意点:**
- 削除された患者が再度 LIFF アプリにアクセスした場合、**完全に新規ユーザー**として扱われます
- 以前のスタンプ履歴、特典交換履歴、ケア記録は**復元されません**
- `line_user_id` は同じですが、`profiles.id` も同じなので、過去のログ（`event_logs`）との紐付けは可能です

LIFF 開発者は以下のテストを実施してください：

- [ ] 新規ユーザーがアクセスした場合、オンボーディング画面が表示される

- [ ] 削除されたユーザーが再度アクセスした場合、オンボーディング画面が表示される
- [ ] 再登録後、以前のデータ（スタンプ、履歴）は引き継がれない（新規状態）
- [ ] エラー発生時、適切なエラーメッセージが表示される

## 注意事項とリスク

### ⚠️ 重大なリスク

#### 1. 誤削除のリスク
**リスク:** 間違った患者を削除してしまう

**対策:**
- ✅ 確認ダイアログで患者名・診察券番号を明示
- ✅ 「削除する」という文字列の入力を必須化
- ✅ 削除ボタンを赤色（destructive）にして目立たせる
- ✅ 削除前に `activity_logs` に詳細情報を記録（監査証跡）

#### 2. データ復元不可のリスク
**リスク:** 削除後、データを復元できない

**対策:**
- ✅ 削除前に `activity_logs` に JSON 形式で詳細情報を保存
- ✅ 確認ダイアログで「この操作は取り消せません」と明示
- 🔄 将来的に、削除前のデータエクスポート機能を提供（オプション）

#### 3. 医療記録削除のリスク
**リスク:** 治療履歴（`patient_dental_records`）が削除される

**対策:**
- ⚠️ 医療記録の保存義務を考慮し、削除前にエクスポート機能を提供（推奨）
- ✅ 削除前に `activity_logs` にケア記録の要約を保存
- 🔄 将来的に、ケア記録のみ残す「プロフィール非表示」機能を検討

#### 4. 家族構成への影響
**リスク:** 代表者を削除した場合、家族構成が変わる

**対策:**
- ✅ 削除前に新代表者を自動選出
- ✅ 確認ダイアログで家族情報を表示
- ✅ 削除後の家族構成を `activity_logs` に記録

---

### 📝 運用上の推奨事項

#### 1. 削除前の確認手順
1. 患者情報を確認（本当に削除対象か？）
2. 関連データ件数を確認（削除される情報を把握）
3. 家族構成を確認（他のメンバーへの影響を把握）
4. 必要に応じて、スタンプ履歴・ケア記録をエクスポート

#### 2. 削除後の対応
1. `activity_logs` で削除ログを確認
2. 家族構成が変わった場合、他のメンバーに通知（オプション）

#### 3. 定期的なバックアップ
- Supabase の自動バックアップ機能を有効化
- 週次または日次でデータベース全体のバックアップを取得
